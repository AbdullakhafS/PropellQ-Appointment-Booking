# US-070: Implement AES-256 Database Encryption

**Story ID:** US-070  
**Epic:** EP-007 (Compliance & Audit Logging)  
**Priority:** CRITICAL  
**Story Points:** 5  
**Owner Role:** Security / Database Engineer  
**Status:** Draft

## User Story

As a **security engineer**, I want all patient and audit data encrypted at rest with AES-256 so that PHI is protected even if storage is compromised.

## Business Value

- Meets HIPAA data-at-rest encryption requirements
- Reduces risk of data exposure from compromised storage
- Increases trust for sensitive healthcare data handling

## Scope

In scope:
- Database encryption for PHI data stores
- Encryption of audit logs at rest
- Key management integration with secure key store
- Verification that backups remain encrypted

Out of scope:
- End-to-end encryption of application payloads (handled by TLS)
- Client-side encryption of form data

## Acceptance Criteria

1. Given PHI data stores, when stored on disk, then data is encrypted using AES-256.
2. Given audit logs at rest, then logs are encrypted with AES-256.
3. Given backups are taken, then backup files remain encrypted.
4. Given key management policy is reviewed, then encryption keys are managed by a secure key store.
5. Given a compliance audit, then encryption configuration and evidence is available.

## Technical Notes

- Use database-native TDE or disk-level encryption based on platform.
- Ensure key rotation is supported by the key store.
- Store encryption metadata separately from encrypted data.

## Dependencies

- Infrastructure key management service.
- Audit log storage architecture.

## Definition of Done

- AES-256 encryption enabled for relevant databases and logs.
- Backup encryption verified.
- Key management integration validated.
- Compliance evidence documented.
