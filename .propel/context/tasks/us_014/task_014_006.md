# Task 006: End-to-End Testing & Performance Validation

**Task ID:** task_014_006  
**User Story:** US-014 (Enable Intake Mode Switching - AI ↔ Manual)  
**Epic:** EP-002  
**Priority:** HIGH  
**Story Points:** 1  
**Owner Role:** QA / Frontend Developer  
**Status:** Not Started

## Objective

Conduct comprehensive end-to-end testing of mode switching functionality, validating data consistency, performance, and user experience across all scenarios.

## Requirements

1. **End-to-End Test Scenarios**

   **Scenario 1: Chatbot → Form Switch**
   - Start intake in AI mode
   - Complete chatbot: chief complaint, medical history, medications, allergies, insurance
   - Click "Switch to Form"
   - Verify: All form fields pre-populated with chatbot data
   - Edit: Modify one medication (name, dosage, frequency)
   - Submit: Form submits with modified data
   - Verify: Submitted data includes both chatbot and edited data

   **Scenario 2: Form → Chatbot Switch**
   - Start intake in form mode
   - Fill form: Chief complaint, select 3 conditions, add 2 medications
   - Click "Switch to Chatbot"
   - Verify: Chatbot displays: "I see you have [conditions]. Let me ask follow-up questions."
   - Continue: Complete chatbot (insurance, allergies, etc.)
   - Submit: Chatbot submits complete intake

   **Scenario 3: Chatbot → Form → Chatbot (Multiple Switches)**
   - Start in chatbot mode, progress to medical history
   - Click "Switch to Form" → form loads with partial data
   - Fill form sections not yet in chatbot (insurance)
   - Click "Switch to Chatbot" (should show warning: "last switch")
   - Verify: Chatbot has all form data + previous chatbot data
   - Submit: Complete intake has all data from both modes

   **Scenario 4: Third Switch Attempt (Should Fail)**
   - Complete scenario 3 (2 switches used)
   - Try to switch back → Button disabled or error shown
   - Message: "You've used your available mode switches"
   - Proceed to submit in current mode

   **Scenario 5: Session Resume (Page Refresh)**
   - Start intake, fill form partially (chief complaint + 2 conditions)
   - Close browser or refresh page
   - Navigate back to intake URL
   - Verify: Resume prompt appears: "You have a saved intake in progress"
   - Click "Yes, Resume"
   - Verify: Form reloaded with all previously entered data
   - Continue: Fill remaining sections and submit

   **Scenario 6: Network Error on Switch**
   - Start in form mode, begin filling data
   - Click "Switch to Chatbot" but simulate network error (API returns 500)
   - Verify: Error message shown: "Could not switch modes. Retry? [Yes] [No]"
   - Click "Retry"
   - Verify: Switch succeeds on retry
   - Complete intake

   **Scenario 7: Data Consistency Across Switches**
   - Start in chatbot mode, enter medication: "Metformin 500mg 2x daily"
   - Switch to form → verify medication appears in repeating fields
   - Edit medication: "Metformin 1000mg 1x daily"
   - Switch back to chatbot → verify chatbot shows updated medication
   - Submit: Submitted data shows final edited value

   **Scenario 8: Mobile Flow**
   - Test scenario 1 on mobile (375px viewport)
   - Verify: Switch button visible without scrolling
   - Verify: Modal responsive and readable on mobile
   - Verify: Form fields editable on mobile
   - Verify: Touch targets 44px+ (buttons, inputs)

   **Scenario 9: Accessibility (Keyboard Only)**
   - Test scenario 1 using only keyboard (no mouse)
   - Tab to "Switch to Form" button → press Enter
   - Tab through modal buttons → press Enter to switch
   - Tab through form fields → fill data
   - Submit form using keyboard
   - Verify: All interactions possible without mouse

   **Scenario 10: Accessibility (Screen Reader)**
   - Test scenario 1 with screen reader enabled (NVDA, JAWS, VoiceOver)
   - Verify: Mode options announced ("AI Assistant option, Fill Out Form option")
   - Verify: Switch button announced with current count ("Switch to Form, 1 more switch available")
   - Verify: Data summary in modal announced
   - Verify: Form labels announced
   - Verify: Error messages announced

2. **Data Consistency Validation**
   - Chief complaint: Should match across modes (no truncation, special characters preserved)
   - Medical history: Conditions should map correctly (checkboxes → chatbot context → form checkboxes)
   - Medications: Array structure preserved, no duplication
   - Allergies: Drug allergies list preserved, reactions maintained
   - Insurance: All fields (name, member ID, group number) preserved
   - Test with: Unicode characters, long text (500 chars), multiple spaces, special characters

3. **Performance Benchmarks**
   - Mode switch time: <1 second (from button click to new component rendered)
   - Draft save: <500ms
   - Draft load: <300ms
   - Data mapping: <100ms
   - Form pre-population: <500ms (from mapped data to visible form)
   - Chatbot resume: <1 second (render with context)

4. **Load Testing**
   - 100 concurrent users switching modes simultaneously
   - 1000 concurrent draft saves
   - Verify: No data loss, no errors
   - Monitor: Server response times, database performance

5. **Browser Compatibility**
   - Chrome (latest)
   - Firefox (latest)
   - Safari (latest)
   - Edge (latest)
   - iOS Safari (mobile)
   - Android Chrome (mobile)

6. **Error Scenarios**
   - Network timeout on draft save: Graceful fallback, allow continue
   - Invalid appointment ID: Show error, redirect to home
   - Expired draft (>24 hours): Don't show resume prompt, start fresh
   - Data mapping failure: Show error, allow user to continue or start fresh
   - Missing required fields in mapped data: Validate and warn user

7. **Regression Testing**
   - Ensure existing chatbot (US-011) functionality unchanged
   - Ensure existing form (US-013) functionality unchanged
   - Verify: No breaking changes to API contracts
   - Test: Previous intake submissions still work
