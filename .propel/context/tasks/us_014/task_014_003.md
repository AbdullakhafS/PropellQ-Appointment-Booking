# Task 003: Build Mode Switch UI and Confirmation Modal

**Task ID:** task_014_003  
**User Story:** US-014 (Enable Intake Mode Switching - AI ↔ Manual)  
**Epic:** EP-002  
**Priority:** CRITICAL  
**Story Points:** 2  
**Owner Role:** Frontend Developer (React)  
**Status:** Not Started

## Objective

Implement UI components for mode switching including a prominent switch button and confirmation modal with data preservation messaging.

## Requirements

1. **Mode Selection Screen**
   - Display at intake start if mode not already selected
   - Title: "How would you prefer to complete intake?"
   - Two clear buttons: [Use AI Assistant] [Fill Out Form]
   - Optional: Short description under each option
   - Default to AI if patient doesn't select (with consent checkbox)
   - Mobile responsive: Stacked buttons on mobile

2. **Switch Button (Always Visible)**
   - Location: Fixed or sticky at top or bottom of intake UI
   - Text: "Switch to Form?" (when in AI mode) or "Switch to Chatbot?" (when in form mode)
   - Styling: High contrast, distinct color (e.g., button color different from primary)
   - Size: 44px+ height on mobile, clearly tappable
   - Icon: Optional switch icon or ">" arrow to indicate action
   - Visible on both desktop and mobile without scrolling (or sticky)

3. **Confirmation Modal**
   - Trigger: When user clicks switch button
   - Title: "Switch to [other mode]?"
   - Message: "Your responses will be preserved and automatically transferred to the [new mode]."
   - Data preservation reassurance: "All your answers so far: [summary]"
   - Two buttons: [Yes, Switch] [Cancel]
   - Close button (X) defaults to Cancel
   - Modal overlay with focus trap (accessibility)

4. **Switch Summary in Modal**
   - Display what data will be transferred:
     - "Chief complaint: [brief text]"
     - "Medical history: [X conditions]"
     - "Medications: [X items]"
     - "Insurance: [collected / not started]"
   - Keep summary brief (2-3 lines max)

5. **Switch Limit UI**
   - After first switch: Display warning "You have 1 more mode switch available"
   - After second switch: Disable switch button with tooltip "You've used your mode switches"
   - Message appears near switch button (not intrusive)

6. **Post-Switch Feedback**
   - After switching: Show toast/notification "Mode switched! Your data has been transferred."
   - Notification auto-closes after 3 seconds
   - Error handling: If switch fails, show error toast with retry option

7. **Mobile Responsiveness**
   - Switch button visible without horizontal scrolling
   - Modal full-width on mobile with padding
   - Buttons stack vertically on narrow screens
   - Touch targets minimum 44px
   - No overlapping of switch button and form fields

8. **Desktop Responsiveness**
   - Switch button positioned consistently (top-right or bottom-right)
   - Modal centered on screen
   - Buttons side-by-side on desktop
   - Clear visual hierarchy

9. **Accessibility**
   - Switch button has proper aria-label
   - Modal has role="dialog", aria-labelledby, aria-describedby
   - Focus trap in modal (Tab cycles within modal)
   - Keyboard accessible: Tab to buttons, Enter/Space to activate
   - Screen reader announces data summary
   - Warning about remaining switches announced

10. **Loading State**
    - If switch involves async API call (draft save), show loading state
    - Disable buttons during loading
    - Show spinner or progress indicator

## Implementation Steps

1. Create ModeSelection component (initial mode choice)
2. Create SwitchButton component (always-visible button)
3. Create SwitchConfirmationModal component
4. Integrate with IntakeContext (access mode, data, switchCount)
5. Implement modal open/close logic
6. Implement data summary generation
7. Implement switch limit warning
8. Integrate data mapping functions (mapChatbotToForm, mapFormToChatbot)
9. Implement post-switch feedback (toast notification)
10. Test on mobile (375px, 480px), tablet (768px), desktop (1024px+)
11. Test keyboard navigation and screen reader
12. Write component unit tests
