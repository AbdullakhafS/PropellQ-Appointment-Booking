# Task 003: Integrate Walk-In Appointment Booking with Explicit Walk-In Flag

**Task ID:** task_031_003  
**User Story:** US-031 (Build Walk-In Booking UI - Search/Create Patient)  
**Epic:** EP-004  
**Priority:** HIGH  
**Story Points:** 2  
**Owner Role:** Backend Developer  
**Status:** Done

## Objective

Implement booking submission logic that persists walk-in appointments with explicit walk-in context and routes the record to queue visibility.

## Requirements

1. Booking payload integration:
   - Include selected/created patient ID
   - Include provider/time slot details
   - Persist `is_walk_in = true` (or equivalent flag)

2. Workflow behavior:
   - Support booking continuation from search or create flow
   - Validate required booking fields prior to submit

3. Persistence and queue readiness:
   - Appointment record is saved with walk-in flag
   - Record appears in staff queue feed with walk-in indicator

4. Error handling:
   - Booking failure surfaces clear message and retry action
   - Preserve selected patient and entered booking data on failure

## Implementation Steps

1. Update booking request contract to include walk-in flag.
2. Wire UI submit action to booking API.
3. Ensure backend stores and returns walk-in context.
4. Integrate queue visibility hook or refresh event.
5. Add integration tests for booking persistence and queue appearance.

## Acceptance Criteria

- [x] Booking submits successfully for selected or newly created patient
- [x] Appointment record includes walk-in flag
- [x] Walk-in appointment becomes visible in staff queue
- [x] Booking errors are recoverable without data loss

## Dependencies

- Appointment booking APIs
- Queue management endpoints/views
- Task 031_001 and Task 031_002 completion

## Validation

- [x] Booking payload verified in API logs
- [x] Queue entry displays walk-in status correctly
- [x] Retry flow tested after simulated API failure

## Definition of Done

- Walk-in booking persistence and queue linkage are working reliably.
