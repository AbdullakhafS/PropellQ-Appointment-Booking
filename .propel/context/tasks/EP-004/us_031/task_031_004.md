# Task 004: Build Walk-In Confirmation Summary Screen

**Task ID:** task_031_004  
**User Story:** US-031 (Build Walk-In Booking UI - Search/Create Patient)  
**Epic:** EP-004  
**Priority:** HIGH  
**Story Points:** 1  
**Owner Role:** Frontend Developer  
**Status:** Done

## Objective

Display clear booking confirmation after walk-in submission, including patient, provider, time, and walk-in status.

## Requirements

1. Summary content:
   - Patient full name
   - Assigned provider
   - Appointment date/time
   - Walk-in status badge/label
   - Booking reference ID if available

2. UX actions:
   - Option to return to queue view
   - Option to start another walk-in booking

3. State handling:
   - Show confirmation only after successful booking response
   - Avoid duplicate submit and duplicate booking creation

4. Accessibility:
   - Proper heading hierarchy and focus management on summary load

## Implementation Steps

1. Build confirmation view component.
2. Map booking response payload to summary fields.
3. Add post-booking action buttons.
4. Add disabled submit protection during in-flight request.
5. Add UI tests for summary rendering.

## Acceptance Criteria

- [x] Confirmation screen appears on successful booking
- [x] Summary includes patient, provider, time, and walk-in status
- [x] Staff can navigate to queue or new booking from summary
- [x] Duplicate submission prevention is in place

## Dependencies

- Task 031_003 booking API integration
- Queue route in staff portal

## Validation

- [x] Confirmation details match saved appointment
- [x] Focus lands on summary heading after transition
- [x] Repeat-booking guard tested

## Definition of Done

- Walk-in confirmation summary is implemented and staff-ready.
