# Task 001: Implement Patient Search Experience for Walk-In Booking

**Task ID:** task_031_001  
**User Story:** US-031 (Build Walk-In Booking UI - Search/Create Patient)  
**Epic:** EP-004  
**Priority:** CRITICAL  
**Story Points:** 2  
**Owner Role:** Frontend Developer  
**Status:** Not Started

## Objective

Build a fast patient search interface for front-desk staff that supports lookup by name, phone, email, and patient ID with incremental results.

## Requirements

1. Search input behavior:
   - Single search field with placeholder guidance for supported identifiers
   - Incremental search with debounce (250-400ms)
   - Trigger search after minimum input length threshold

2. Search result list:
   - Display matching patient rows with key identifiers
   - Highlight primary match fields (name, DOB, phone, patient ID)
   - Support keyboard navigation and click selection

3. Empty and error states:
   - No matches: prompt to create new patient
   - API error: show retry state with non-blocking message

4. Performance target:
   - Results appear quickly for normal desk workflow (<500ms API target)

## Implementation Steps

1. Build walk-in search component UI.
2. Wire incremental search to patient lookup API.
3. Add result list rendering and selection handler.
4. Implement no-match and error states.
5. Add component tests for search interactions.

## Acceptance Criteria

- [ ] Staff can search by name, phone, email, or patient ID
- [ ] Matching records appear with incremental search behavior
- [ ] Search results can be selected to continue booking
- [ ] No-match state clearly offers create-patient path
- [ ] UI handles API failure gracefully

## Dependencies

- Patient service APIs from EP-001/EP-002
- Staff portal shell and routing

## Validation

- [ ] Search tests cover all supported input types
- [ ] Debounce behavior verified
- [ ] No-result flow verified

## Definition of Done

- Walk-in search UI is implemented and ready for staff usage.
