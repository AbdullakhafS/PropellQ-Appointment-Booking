# Task 002: Expose Walk-In Status in Queue and Schedule APIs

**Task ID:** task_033_002  
**User Story:** US-033 (Flag Walk-In Appointments)  
**Epic:** EP-004  
**Priority:** HIGH  
**Story Points:** 1  
**Owner Role:** Backend Developer  
**Status:** Done

## Objective

Ensure queue and schedule API responses include walk-in status so presentation layers can render a consistent indicator.

## Requirements

1. API contract updates:
   - Include `is_walk_in` in schedule list responses
   - Include `is_walk_in` in queue list responses

2. Consistency:
   - Field naming and type consistent across endpoints
   - Include in both summary and detail payloads if applicable

3. Backward compatibility:
   - Avoid breaking consumers expecting existing response fields

## Implementation Steps

1. Update DTO/serializer for queue and schedule responses.
2. Map appointment walk-in field into API payloads.
3. Add API tests for walk-in and non-walk-in records.
4. Update API documentation/spec examples.

## Acceptance Criteria

- [x] Queue API includes walk-in status field
- [x] Schedule API includes walk-in status field
- [x] Responses are consistent across endpoints
- [x] API tests cover true/false walk-in states

## Dependencies

- Task 033_001 data model update
- Queue and schedule endpoint ownership

## Validation

- [x] API payload inspection confirms walk-in field presence
- [x] Existing clients continue functioning after update

## Definition of Done

- Walk-in status is exposed reliably through queue/schedule APIs.
