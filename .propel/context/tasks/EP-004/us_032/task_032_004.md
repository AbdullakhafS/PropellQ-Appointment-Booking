# Task 004: Implement No-Availability Fallback Messaging and Alternatives

**Task ID:** task_032_004  
**User Story:** US-032 (Assign Walk-In to Available Slots)  
**Epic:** EP-004  
**Priority:** HIGH  
**Story Points:** 1  
**Owner Role:** Frontend/UX Developer  
**Status:** Done

## Objective

Provide clear fallback behavior when no slots are available, including actionable alternatives for front-desk staff.

## Requirements

1. No-slot messaging:
   - Show clear message when provider/clinic is fully booked
   - Avoid generic error wording

2. Alternative options:
   - Suggest switching provider
   - Suggest switching clinic or widening time window
   - Option to place patient in waitlist-like workflow if supported

3. UX flow:
   - Keep user in walk-in flow without full reset
   - Provide quick controls to retry availability lookup

4. Accessibility:
   - Alert region and focus handling for no-availability state

## Implementation Steps

1. Build no-availability state component.
2. Add provider/clinic alternative actions.
3. Integrate with slot lookup refresh logic.
4. Add tests for fallback UX behavior.

## Acceptance Criteria

- [x] Clear no-slot message is shown when fully booked
- [x] Staff can take alternative actions without leaving flow
- [x] Retry path works and refreshes availability
- [x] Accessibility behavior validated

## Dependencies

- Task 032_001 and task_032_002
- Provider/clinic selector controls

## Validation

- [x] Fully booked test case shows fallback UI
- [x] Alternative actions trigger new availability lookup
- [x] No dead-end states remain in booking UI

## Definition of Done

- No-availability handling is actionable and user-friendly.
