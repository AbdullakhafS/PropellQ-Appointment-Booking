# US-051: Implement Session Token with Role Info

**Story ID:** US-051  
**Epic:** EP-005 (User Access & Role Management)  
**Priority:** CRITICAL  
**Story Points:** 5  
**Owner Role:** Authentication Engineer  
**Status:** Draft

## User Story

As a **platform engineer**, I want session tokens to include user role information so that authorization checks can be performed efficiently without additional role lookups on every request.

## Business Value

- Reduces authorization latency and overhead
- Enables centralized token-based access control
- Supports consistent enforcement of user permissions

## Scope

In scope:
- Include role and relevant claims in authentication tokens
- Define token payload structure and expiration rules
- Validate token roles during authorization checks
- Avoid sensitive data in token payloads

Out of scope:
- Token refresh mechanism beyond role claim validation
- External SSO provider integrations

## Acceptance Criteria

1. Given a user authenticates, then the issued session token contains the user’s role and relevant access claims.
2. Given a request includes a valid token, then the authorization layer can resolve permissions from token claims.
3. Given a token is expired or invalid, then the system rejects the request with a 401 Unauthorized.
4. Given role changes occur, then tokens are invalidated or refreshed to reflect updated permissions.
5. Given token payloads are examined, then no sensitive PHI is stored in the token.

## Technical Notes

- Consider JWT or secure opaque token format with role claims.
- Keep payload minimal and do not store patient identifiers if not needed.
- Implement token revocation or short-lived tokens if role changes are common.

## Dependencies

- Authentication service from EP-005.
- RBAC model definitions from US-043.

## Definition of Done

- Session tokens carry role information and are validated on every request.
- Token invalidation strategy in place for role changes.
- Security review ensures no PHI leaked in tokens.
- Tests validate token claims and authorization flows.
