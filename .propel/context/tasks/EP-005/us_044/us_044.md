# US-044: Define Patient Role & Permissions

**Story ID:** US-044  
**Epic:** EP-005 (User Access & Role Management)  
**Priority:** HIGH  
**Story Points:** 3  
**Owner Role:** Product / Security Engineer  
**Status:** Draft

## User Story

As a **product owner**, I want the Patient role to have access only to their own appointment, intake, and profile data so that patient privacy is maintained and the user experience is secure.

## Business Value

- Protects patient privacy and PHI
- Builds patient trust through scoped access
- Enables a straightforward self-service patient experience

## Scope

In scope:
- Patient access to appointment booking, reminders, intake forms, and dashboard
- Restrict patient access to only their own records
- Patient-facing profile and notification preference management

Out of scope:
- Staff and admin permissions
- External provider portal access

## Acceptance Criteria

1. Given a Patient user is authenticated, when they request their appointments, then the system returns only that patient's appointments.
2. Given a Patient user requests profile or intake data, then only their own records are returned.
3. Given a Patient user attempts to access another patient's appointment or intake data, then the system returns 403 Forbidden.
4. Given a Patient user accesses the dashboard, then they only see data relevant to their own account.
5. Given a Patient user interacts with notifications, then they cannot modify another user's notification preferences.

## Technical Notes

- Use user ID scoping in all patient-facing endpoints.
- Avoid exposing patient list or search capability to patient role.
- Enforce ownership checks at the service layer.

## Dependencies

- User authentication and session model from EP-005.
- RBAC permission model defined in US-043.

## Definition of Done

- Patient role permissions implemented and validated.
- Ownership checks in place on all patient-facing APIs.
- Role permissions documented and reviewed.
- Regression tests cover unauthorized access attempts.
