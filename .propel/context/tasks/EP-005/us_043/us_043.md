# US-043: Implement RBAC Permission Model

**Story ID:** US-043  
**Epic:** EP-005 (User Access & Role Management)  
**Priority:** CRITICAL  
**Story Points:** 5  
**Owner Role:** Security / Platform Engineer  
**Status:** Draft

## User Story

As an **admin architect**, I want to define a role-based access control model for Patient, Staff, and Admin personas so that system authorization is enforced consistently and sensitive data access is restricted by role.

## Business Value

- Ensures least-privilege access for different user personas
- Reduces risk of unauthorized disclosures of PHI
- Enables compliance with HIPAA access control requirements

## Scope

In scope:
- Role definitions and permissions matrix for Patient, Staff, and Admin
- Backend authorization middleware or policy enforcement
- Role-aware API responses and UI gating
- Permission checklist for core appointment, intake, queue, and audit flows

Out of scope:
- Full user management UI (handled in US-047/US-048)
- Advanced RBAC delegation or multi-tenant role inheritance

## Acceptance Criteria

1. Given an authenticated user, when they request a protected resource, then role-based permission checks are executed before returning data.
2. Given a Patient user, when they access appointment or profile endpoints, then they only see their own records and not other patients'.
3. Given a Staff user, when they access queue or check-in endpoints, then they can access only assigned patient appointments and not unrelated patient details.
4. Given an Admin user, when they access management endpoints, then they have full access to user account and audit resources.
5. Given a request fails authorization, then the API returns a standard 403 response and logs the denied action.
6. Given role definitions are updated, then the permission model is documented and reviewable.

## Technical Notes

- Map permissions to specific API actions, not just broad roles.
- Use claims-based authorization in tokens to avoid repeated role lookups.
- Implement permission checks in middleware or service layer for reuse.

## Dependencies

- EP-005 technical design and policy definitions.
- Authentication/session token model from EP-005.

## Definition of Done

- RBAC model implemented and tested for all core role-based endpoints.
- Permission failures handled gracefully with clear 403 semantics.
- RBAC rules documented in architecture docs.
- Peer-reviewed and security-approved.
