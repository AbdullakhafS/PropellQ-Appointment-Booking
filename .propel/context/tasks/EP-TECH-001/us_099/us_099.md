# US-099: Implement Centralized Logging with Correlation IDs

**Story ID:** US-099  
**Epic:** EP-TECH-001 (Technical Foundation & DevEx)  
**Priority:** CRITICAL  
**Story Points:** 5  
**Owner Role:** Platform Engineer / SRE  
**Status:** Draft

## User Story

As an SRE, I want centralized structured logs with correlation IDs across services so that incidents can be diagnosed quickly and reliably.

## Business Value

- Reduces mean time to recovery (MTTR) by enabling end-to-end request traceability.
- Improves compliance and audit readiness through consistent log structure.
- Simplifies operational support and production debugging.

## Scope

In scope:
- JSON structured logging standard with required fields.
- Correlation ID propagation across API boundary and async workers.
- Centralized log shipping pipeline and searchable index.
- PII/PHI masking policy for application logs.
- Standard log levels and retention profile by environment.

Out of scope:
- Full SIEM analytics and long-term security data lake design.

## Acceptance Criteria

1. Given an incoming request without correlation ID, when processed, then the gateway/service generates one and propagates it through downstream calls.
2. Given a request spans multiple services, when logs are queried by correlation ID, then all related log events are discoverable in one timeline.
3. Given logs include sensitive fields, when events are emitted, then masked/redacted rules prevent PHI/secret leakage.
4. Given production traffic, when logs are ingested, then log delivery success rate is >= 99.9% with retry on transient failures.
5. Given an incident investigation, when searching centralized logs, then engineers can filter by service, environment, severity, and correlation ID.
6. Given retention policy by environment, when retention windows are applied, then logs are retained and expired per policy.

## Technical Notes

- Target requirements: NFR-005, NFR-007; TR-006, TR-008.
- Required fields: timestamp, service, env, severity, correlationId, actorId (if available), route, statusCode.
- Integrate with immutable audit logging boundaries from compliance epic where applicable.

## Dependencies

- API middleware contract from US-098 for correlation ID injection.
- Infrastructure sinks available for aggregation.

## Risks and Mitigations

- Risk: High log volume increases cost.
- Mitigation: Sampling strategy for debug logs and tiered retention.

## Definition of Done

- Logging library/package integrated in core services.
- Correlation IDs present in > 95% of service logs in staging tests.
- Redaction rules verified with security checks.
- Runbook updated for incident log investigations.

## INVEST Check

- Independent: Yes, can be delivered as platform capability.
- Negotiable: Yes, sink/vendor details can vary.
- Valuable: Yes, direct reliability and operability gain.
- Estimable: Yes, bounded by service pilot + pipeline.
- Small: Yes, sprint-sized with pilot services.
- Testable: Yes, through propagation, searchability, and redaction tests.
