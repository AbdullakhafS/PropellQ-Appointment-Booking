# US-102: Implement Resiliency Defaults (Timeouts, Retries, Circuit Breakers)

**Story ID:** US-102  
**Epic:** EP-TECH-001 (Technical Foundation & DevEx)  
**Priority:** CRITICAL  
**Story Points:** 8  
**Owner Role:** Backend Engineer / SRE  
**Status:** Draft

## User Story

As a backend engineer, I want default resiliency patterns for service-to-service communication so that transient failures are contained and the platform degrades gracefully instead of cascading.

## Business Value

- Prevents cascading failures during partial outages.
- Improves user-facing availability and response stability.
- Establishes safe-by-default service communication patterns.

## Scope

In scope:
- Standard timeout defaults by call type (sync/async/external).
- Retry policy with jitter and retry budgets.
- Circuit breaker policy with half-open recovery behavior.
- Fallback behavior guidelines for non-critical dependencies.

Out of scope:
- Re-architecture of all downstream dependencies.

## Acceptance Criteria

1. Given outbound service calls, when no explicit timeout is set, then safe default timeout values apply by endpoint type.
2. Given transient errors (5xx, timeout), when retries are attempted, then exponential backoff with jitter is applied and capped by retry budget.
3. Given repeated downstream failures, when threshold is exceeded, then circuit breaker opens and prevents further pressure on failing dependency.
4. Given circuit is open, when probe interval is reached, then half-open behavior tests recovery before closing.
5. Given non-critical dependency outage, when fallback path exists, then core booking flow remains available with degraded non-critical features.
6. Given resiliency behavior, when simulated chaos tests run, then system does not exceed defined error-rate/latency guardrails.

## Technical Notes

- Target requirements: NFR-001, NFR-002, NFR-003; TR-006, TR-007, TR-008.
- Package resiliency defaults into shared client wrapper used by all services.

## Dependencies

- Shared API/middleware baseline from US-098.
- Observability and alerting support from US-100.

## Risks and Mitigations

- Risk: Incorrect retry policy can amplify load.
- Mitigation: Per-endpoint retry allowlist and retry budget caps.

## Definition of Done

- Shared resiliency library integrated in core services.
- Chaos and load tests validate breaker/retry/timeout behavior.
- Operational runbook includes breaker states and recovery actions.
- Fallback behavior documented for critical journeys.
