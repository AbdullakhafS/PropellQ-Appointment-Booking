# US-089: Implement Async Queue for Reminders

**Story ID:** US-089  
**Epic:** EP-008 (System Reliability & Performance)  
**Priority:** HIGH  
**Story Points:** 5  
**Owner Role:** Backend / SRE Engineer  
**Status:** Draft

## User Story

As a **backend engineer**, I want appointment reminders to be processed via an asynchronous queue so that booking flows are non-blocking and reminder delivery can scale independently.

## Business Value

- Improves booking responsiveness by offloading reminder jobs
- Supports scalable delivery for email and SMS reminders
- Enables retry and failure handling without blocking user requests

## Scope

In scope:
- Queued reminder job creation after appointment booking
- Worker processes for email/SMS reminder delivery
- Retry and dead-letter handling for failed reminder jobs

Out of scope:
- Reminder content authoring UI
- External SMS provider configuration beyond queue integration

## Acceptance Criteria

1. Given a booking event, then reminder jobs are enqueued asynchronously.
2. Given a worker processes jobs, then email and SMS reminders are sent reliably.
3. Given a job fails, then retry logic applies and failed messages are moved to dead-letter handling.
4. Given reminder delivery status is requested, then job state is queryable.

## Technical Notes

- Use a durable queue system such as Redis Streams, RabbitMQ, or cloud queue.
- Ensure jobs are idempotent to prevent duplicate reminders.
- Capture delivery status and failure reasons for monitoring.

## Dependencies

- Booking event or appointment creation flow from EP-001.
- Notification delivery infrastructure.

## Definition of Done

- Reminder queue implemented and validated.
- Retry and dead-letter handling configured.
- Monitoring of queue processing enabled.
- Integration tests verify asynchronous behavior.
