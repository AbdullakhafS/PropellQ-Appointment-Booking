# US-020: Implement Document Upload & Processing

**Story ID:** US-020  
**Epic:** EP-003 (Clinical Data Intelligence & Coding)  
**Priority:** CRITICAL  
**Story Points:** 5  
**Owner Role:** Backend / Document Processing Engineer  
**Status:** Draft

## User Story

As a **clinical staff member**, I want to upload patient documents and have them processed automatically so that extracted clinical data is available without manual transcription.

## Business Value

- Reduces manual data entry and transcription errors
- Speeds clinical preparation and intake review
- Captures structured information from unstructured documents

## Scope

In scope:
- Document upload support for PDF and DOCX
- Processing pipeline to extract text and metadata
- Validation of successful upload and processing status

Out of scope:
- Document annotation or manual correction UI

## Acceptance Criteria

1. Given a document upload, then the system accepts PDF or DOCX files and stores them securely.
2. Given a processed document, then extracted metadata is available for downstream use.
3. Given an unsupported file type, then the upload is rejected with an informative error.
4. Given document processing completes, then status is updated and accessible from the patient profile.

## Technical Notes

- Store uploaded files securely with PHI protections.
- Use a processing job queue to handle file extraction asynchronously.
- Retain original document references for traceability.

## Dependencies

- Secure storage and intake profile integration.
- Processing worker or external document extraction service.

## Definition of Done

- PDF and DOCX uploads work end-to-end.
- Documents are processed and status tracked.
- Extracted metadata is stored for profile aggregation.
- Tests verify upload, rejection, and processing flows.
