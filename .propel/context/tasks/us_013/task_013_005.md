# Task 005: Implement Auto-Population Logic & Consent Flow

**Task ID:** task_013_005  
**User Story:** US-013 (Build Manual Intake Form with Auto-Population)  
**Epic:** EP-002  
**Priority:** CRITICAL  
**Story Points:** 3  
**Owner Role:** Frontend Developer (React)  
**Status:** Not Started

## Objective

Implement consent flow ("Use Previous" / "Start Fresh") and auto-populate form fields with patient's previous intake data on form load.

## Requirements

1. **Consent Dialog on Form Load**
   - Display modal/dialog on initial form load (before showing form)
   - Title: "We have your previous information"
   - Message: "Would you like us to use it to save time?"
   - Last updated text: "(Last updated: June 15, 2026)"
   - Two buttons: [Use Previous] [Start Fresh]
   - Dialog is dismissible with X button (defaults to "Start Fresh")
   - Consent preference stored in session/localStorage for session duration

2. **API Call for Previous Data**
   - On form load, call `GET /api/patients/{patientId}/intake/last`
   - Handle 404 if no previous data (show "Start Fresh" only)
   - Handle network errors gracefully (retry logic or fallback to "Start Fresh")
   - Loading state while fetching: Show spinner in dialog

3. **Auto-Population Logic**
   - If [Use Previous] clicked:
     - Populate Chief Complaint field with `previous.chief_complaint`
     - Populate Medical History checkboxes based on `previous.medical_history.conditions`
     - Populate Medications array with `previous.medications` entries
     - Populate Allergies array with `previous.allergies` entries
     - Populate Insurance fields with `previous.insurance` values
   - If [Start Fresh] clicked: Show empty form with no pre-filled values
   - Display "Last updated: [date]" text under each section (read-only)

4. **Form State on Load**
   - No validation errors shown on page load (even if data is pre-filled)
   - All fields are editable (patient can immediately modify values)
   - Pre-filled fields show no visual difference (not grayed out)
   - User can delete any pre-filled value without friction

5. **Timestamp Tracking**
   - Display "Last updated: June 15, 2026" in each section
   - Timestamp is fetched from `previous.last_updated_at`
   - Update timestamp only when form is submitted (not on every edit)
   - Format timestamp as: "June 15, 2026" (user-friendly date)

6. **Repeating Fields Auto-Population**
   - Medications array pre-populated with previous medications
   - Each medication retains name, dosage, frequency from previous intake
   - Allergies array pre-populated with previous allergies
   - Each allergy retains name, reaction type, description

7. **Error Handling**
   - Network error fetching previous data: Show error message and "Start Fresh" button
   - Retry button for network errors: "Retry loading your information"
   - After 2 failed retries: Show "Start Fresh" only
   - Log errors to console for debugging

8. **Performance**
   - Load form and fetch previous data in parallel (no sequential delay)
   - Show consent dialog while API call is in progress
   - Once API returns, populate form fields immediately (no flicker)
   - Target: Consent dialog appears within 500ms

9. **Accessibility**
   - Consent dialog has proper ARIA attributes: `role="dialog"`, `aria-labelledby`, `aria-describedby`
   - Dialog has focus trap (Tab cycles within dialog only until dismissed)
   - Buttons are keyboard accessible: Tab, Enter to activate
   - "Last updated" text is readable and doesn't distract from main content

## Implementation Steps

1. Create ConsentDialog component
2. Implement API call to fetch previous intake data on form load
3. Implement loading state in ConsentDialog (spinner while fetching)
4. Implement error handling with retry logic
5. Create useAutoPopulate hook to handle population logic
6. Populate Chief Complaint, Medical History, Insurance sections
7. Populate Medications array using fieldArray.append()
8. Populate Allergies array using fieldArray.append()
9. Display "Last updated" timestamps in sections (read-only)
10. Test with previous data, no previous data, network errors
11. Test on mobile: Dialog is readable and buttons are tappable
12. Write unit tests for auto-population logic and error handling
