# Task 001: Create Backend Auto-Population API

**Task ID:** task_013_001  
**User Story:** US-013 (Build Manual Intake Form with Auto-Population)  
**Epic:** EP-002  
**Priority:** CRITICAL  
**Story Points:** 3  
**Owner Role:** Backend Developer  
**Status:** Not Started

## Objective

Implement backend API endpoint to retrieve patient's previous intake data for form auto-population, with caching and error handling.

## Requirements

1. **API Endpoint**
   - Endpoint: `GET /api/patients/{patientId}/intake/last`
   - Authentication: Verify user is accessing own data or is clinic staff
   - Response format: JSON with chief_complaint, medical_history, medications, allergies, insurance, last_updated_at

2. **Database Query**
   - Query `IntakeResponses` table for most recent completed intake for patient
   - Return last submitted intake regardless of mode (AI or manual)
   - Order by `completed_at DESC` LIMIT 1
   - Include `last_updated_at` timestamp for UI display

3. **Response Schema**
   ```json
   {
     "chief_complaint": "Chest pain",
     "medical_history": {
       "conditions": ["hypertension", "diabetes"],
       "other": "Previous heart surgery 2015"
     },
     "medications": [
       {
         "name": "Metformin",
         "dosage": "500mg",
         "frequency": "2x daily"
       }
     ],
     "allergies": {
       "drug_allergies": ["penicillin"],
       "reactions": { "penicillin": "rash" },
       "side_effects": ["amoxicillin causes GI upset"]
     },
     "insurance": {
       "name": "Aetna",
       "member_id": "12345",
       "group_number": "67890",
       "plan_name": "PPO"
     },
     "last_updated_at": "2026-06-15T14:30:00Z"
   }
   ```

4. **Error Handling**
   - 404 if patient not found
   - 404 if no previous intake data exists (return empty object)
   - 401 if user not authenticated
   - 403 if user not authorized to view patient data
   - 500 for database errors with generic message (log details)

5. **Caching Strategy**
   - Cache response for 15 minutes per patient
   - Invalidate cache when new intake submitted
   - Redis or in-memory cache with TTL

6. **Performance**
   - Response time: <500ms (including cache lookup)
   - Database query optimized: index on (patient_id, completed_at DESC)

## Implementation Steps

1. Design API contract and response schema
2. Create database indexes on IntakeResponses (patient_id, completed_at)
3. Implement query logic: Get last intake for patient, ordered by completed_at
4. Implement cache layer (Redis or in-memory with TTL)
5. Add authentication/authorization checks
6. Add comprehensive error handling and logging
7. Implement response transformation (database model → API DTO)
8. Write integration tests with mock data
9. Load test: 1000 concurrent requests → verify <500ms response
10. Document API endpoint and response schema
