# Task 006: Implement Mobile Responsiveness & Accessibility Audit

**Task ID:** task_013_006  
**User Story:** US-013 (Build Manual Intake Form with Auto-Population)  
**Epic:** EP-002  
**Priority:** HIGH  
**Story Points:** 2  
**Owner Role:** Frontend Developer + QA  
**Status:** Not Started

## Objective

Ensure form is fully responsive across all device sizes (375px to 1920px) and meets WCAG 2.1 Level AA accessibility standards.

## Requirements

1. **Mobile Responsiveness (375px - 480px)**
   - Single-column layout: All form sections stack vertically
   - Input fields: Full width (100% minus padding)
   - Input height: Minimum 44px for touch-friendly tapping
   - Font size: Minimum 16px (prevents auto-zoom on iOS)
   - Button size: Minimum 44px width and height
   - Sticky submit button: Fixed to bottom with safe area padding (for notches)
   - Sticky consent dialog: Full-width modal with safe area margins
   - Section headers: Readable font size (18px+)
   - Error messages: Readable and not cut off on narrow screens

2. **Tablet Responsiveness (480px - 768px)**
   - Two-column layout where appropriate (e.g., Insurance fields: Name and Member ID side-by-side)
   - Wider inputs and buttons for easier tapping
   - Form padding increased (20px+ margins)
   - Better spacing between sections

3. **Desktop Responsiveness (768px - 1024px)**
   - Two-column sections: Chief Complaint (left), Medical History (right)
   - Wider input fields (not consuming full width)
   - Better visual hierarchy with spacing
   - Submit button: Full width or side-by-side with cancel

4. **Large Desktop (1024px+)**
   - Three-column sections or split layouts where appropriate
   - Max-width container (1200px) to prevent overly wide form
   - Sidebar or accordion navigation for sections (optional)

5. **Sticky Submit Button**
   - Mobile: Fixed to bottom, full width, padding above
   - Desktop: Sticky to bottom of form or within form container
   - Always visible and accessible
   - Safe area padding for devices with notches (iPhone X, etc.)

6. **Accessibility - WCAG 2.1 Level AA**
   - **Keyboard Navigation:**
     - All form fields accessible via Tab key
     - Tab order logical and predictable
     - Submit button reachable via Tab
     - Delete/Add buttons accessible via Tab and Enter
   - **Labels:**
     - All input fields have associated `<label>` elements
     - Labels linked via `htmlFor` attribute
     - Required field markers are text ("*") not just visual
   - **Color Contrast:**
     - Text/background contrast ratio ≥4.5:1 (normal text)
     - Button/background contrast ratio ≥4.5:1
     - Error text red has sufficient contrast (not red #FF0000, but #d32f2f or darker)
   - **ARIA Attributes:**
     - Form has `aria-label="Patient Intake Form"`
     - Required fields have `aria-required="true"`
     - Error messages linked via `aria-describedby`
     - Error summary has `role="alert"` and `aria-live="polite"`
     - Disabled submit button has `aria-disabled="true"` if applicable
   - **Focus Visible:**
     - All interactive elements have visible focus indicator (outline or highlight)
     - Focus indicator has sufficient contrast (not color alone)
     - Focus order is logical
   - **Alternative Text:**
     - Icons (if used) have `aria-label` or `title` attribute
     - Delete/Add buttons have descriptive labels, not just icons
   - **Screen Reader Compatibility:**
     - Form sections announced with heading hierarchy (H2, H3)
     - Field labels read correctly by screen reader
     - Error messages announced
     - Required fields indicated verbally ("required", not just "*")

7. **Touch Interaction**
   - All touch targets minimum 44px × 44px
   - Touch targets have adequate spacing (8px between targets)
   - Delete buttons easily tappable (not too close to other buttons)
   - Checkboxes and radio buttons have adequate size and tap area

8. **Orientation Handling**
   - Form works in portrait and landscape on mobile
   - No horizontal scrolling required on any device
   - Safe area padding for landscape mode

## Implementation Steps

1. Review CSS breakpoints (375px, 480px, 768px, 1024px, 1920px)
2. Implement mobile-first CSS approach
3. Test responsive layout at each breakpoint
4. Implement sticky submit button with safe area padding
5. Test on real mobile devices (iPhone 12, Samsung Galaxy A50, iPad)
6. Verify touch-friendly sizes (44px+) at all breakpoints
7. Run accessibility audit tool (axe, WAVE, Lighthouse)
8. Fix color contrast issues (if any)
9. Test keyboard navigation (Tab through entire form)
10. Test with screen reader (NVDA, JAWS, VoiceOver)
11. Verify form labels and ARIA attributes
12. Document accessibility compliance checklist
13. Create screenshots/video demonstrating accessibility
