# US-015: Implement Insurance Pre-Check Logic

**Epic:** EP-002 (Patient Intake & Insurance Verification)  
**Story Points:** 5  
**Priority:** HIGH  
**Status:** Not Started  
**Created:** 2026-06-17  

---

## User Story Statement

As a **billing system**, I want to **perform soft validation of insurance information (name, ID, group number) against an internal predefined database of insurance plans**, so that **unverified insurance is flagged for manual review post-appointment, reducing claim denials and improving billing efficiency**.

---

## Business Value

- **Outcome 1:** Insurance pre-check identifies 70%+ of verification issues before appointment (early warning)
- **Outcome 2:** Reduces claim denials from incorrect insurance data
- **Outcome 3:** Billing team can prioritize high-risk insurances for immediate follow-up

---

## Scope

**In Scope:**
- Soft validation: Check insurance name/ID against predefined database (NOT hard validation with real insurance APIs)
- Database of dummy/test insurance records: Hardcoded list of common insurers (Aetna, Blue Cross, UnitedHealth, etc.)
- Validation rules: Match on insurance name + member ID; flag if name or ID is missing or doesn't match
- Confidence scoring: Return confidence level (0-100%): exact match = 100%, partial match = 50%, no match = 0%
- Flagging system: If confidence < 70%, flag as "unverified" for manual review
- Soft vs. hard: Even if flagged "unverified," allow booking to proceed (soft check, not blocking)
- Audit trail: Log all insurance checks with timestamp, result, confidence score

**Out of Scope:**
- Real-time validation against actual insurance provider APIs (Phase 2)
- Premium verification (coverage verification, copay lookup) — Phase 2
- Claim submission integration — Phase 2

---

## Acceptance Criteria

1. **Insurance database:** Create predefined list of ~50 common insurance plans (Aetna, Anthem Blue Cross, UnitedHealth, Humana, etc.) with typical ID formats; store in database or config file
2. **Data collection:** Intake (chatbot or form) collects: insurance name, member ID, group number (optional), plan name (optional)
3. **Validation trigger:** Upon intake submission, automatically trigger insurance pre-check (no extra user action required)
4. **Name matching:** Check if entered insurance name matches known plan names (case-insensitive, allow partial matches like "Aetna" matches "Aetna Health")
5. **ID format validation:** Check if member ID format matches expected format for that insurance (e.g., Aetna typically 8-10 digits)
6. **Group number validation:** If patient provides group number, validate format (optional field, but validate if present)
7. **Confidence scoring:**
   - Exact match (name + ID format correct) = 100%
   - Partial match (name matches, ID format unclear) = 70%
   - Insurance name recognized, no ID provided = 50%
   - Insurance name unknown = 0%
8. **Flagging:** If confidence < 70%, set flag: insurance_verification_status = "unverified"; if ≥70%, set to "verified"
9. **User notification:** If flagged unverified, display warning to patient: "We couldn't verify your insurance. Don't worry—you can still book. We'll verify it after your visit."
10. **Staff dashboard:** Show pending insurance verifications; allow staff to review and update status manually
11. **Audit logging:** Log all checks: patient_id, insurance_name, member_id, confidence_score, status, timestamp, checked_by (system)

**Technology Stack:**

- API: `POST /api/intake/insurance-check` endpoint

**Database Schema:**
```sql
CREATE TABLE InsurancePlans (
  id INT PRIMARY KEY,
  name VARCHAR (e.g., 'Aetna'),
  aliases JSON (e.g., ['Aetna Health', 'Aetna Insurance']),
  member_id_format VARCHAR (e.g., '^[0-9]{8,10}$' for regex),
  group_number_format VARCHAR (nullable),
  created_at DATETIME
);

CREATE TABLE InsuranceVerifications (
  id INT PRIMARY KEY,
  appointment_id INT FOREIGN KEY,
  patient_id INT FOREIGN KEY,
  provided_insurance_name VARCHAR,
  provided_member_id VARCHAR,
  provided_group_number VARCHAR (nullable),
  matched_plan_id INT FOREIGN KEY (nullable),
  confidence_score INT (0-100),
  verification_status VARCHAR ('verified', 'unverified', 'manual_review'),
  notes VARCHAR (nullable),
  verified_by_staff_id INT (nullable),
  verified_at DATETIME (nullable),
  created_at DATETIME
);

CREATE INDEX idx_insurance_status ON InsuranceVerifications 
  WHERE verification_status = 'unverified';
```

**Insurance Pre-Check Logic (Pseudocode):**
```
FUNCTION checkInsurance(appointment_id, insurance_name, member_id, group_number):
  
  // Step 1: Match insurance name
  matched_plan = FIND_PLAN_BY_NAME(insurance_name)
  IF matched_plan IS NULL:
    confidence = 0
    status = 'unverified'
    RETURN { confidence, status, reason: 'Insurance plan not recognized' }
  
  // Step 2: Validate member ID format
  IF matched_plan.member_id_format AND NOT REGEX_MATCH(member_id, matched_plan.member_id_format):
    confidence = 50  // Name matches, but ID format is off
    status = 'unverified'
  ELSE IF member_id IS NULL:
    confidence = 50  // Name matches, no ID provided
    status = 'unverified'
  ELSE:
    confidence = 100  // Name + ID format match
    status = 'verified'
  
  // Step 3: Validate group number (if provided)
  IF group_number IS NOT NULL AND matched_plan.group_number_format:
    IF NOT REGEX_MATCH(group_number, matched_plan.group_number_format):
      confidence = confidence - 20  // Deduct for invalid group number
      status = 'unverified'
  
  // Step 4: Final status
  IF confidence >= 70:
    status = 'verified'
  ELSE:
    status = 'unverified'
  
  // Step 5: Log in database
  INSERT InsuranceVerifications 
    (appointment_id, patient_id, provided_insurance_name, provided_member_id, 
     matched_plan_id, confidence_score, verification_status, created_at)
  
  RETURN { confidence, status, matched_plan, reason }
```

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Insurance database is outdated or incomplete → misses valid plans | Medium | Maintain insurance plan list; add new plans quarterly; crowdsource invalid matches from staff feedback |
| Regex patterns for member ID are too strict → flags valid IDs as unverified | Medium | Test regex patterns with real member ID samples; collect validation failures from staff; iteratively refine patterns |
| Soft validation gives false sense of security → staff trusts it too much | Low | Document "soft" nature clearly; educate staff that verification is not guaranteed; flag all <70% confidence for manual review |
| Performance: Checking 50 plan names on every intake → slow | Low | Index insurance plans database; use fast string matching algorithms; cache recent checks |
