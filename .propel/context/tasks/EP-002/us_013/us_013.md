# US-013: Build Manual Intake Form with Auto-Population

**Epic:** EP-002 (Patient Intake & Insurance Verification)  
**Story Points:** 8  
**Priority:** HIGH  
**Status:** Not Started  
**Created:** 2026-06-17  

---

## User Story Statement

As a **patient**, I want to **fill out a structured intake form that auto-populates with my previous visit data and allows me to edit fields without friction**, so that **intake takes 3-5 minutes (vs. 20 minutes for new data entry), and I feel confident my information is accurate**.

---

## Business Value

- **Outcome 1:** Auto-population reduces form completion time by 60-70% for returning patients
- **Outcome 2:** Provides alternative for patients who prefer forms over conversational AI (choice increases adoption)
- **Outcome 3:** Structured form captures consistent data for patients who fill it vs. chatbot users

---

## Scope

**In Scope:**
- Multi-section form: chief complaint, medical history, medications, allergies, insurance
- Auto-population: Retrieve patient's previous intake/visit data and pre-fill form fields
- Edit capability: Patient can add, remove, or modify any field without validation errors
- Form validation: Required fields marked; optional fields are truly optional
- Responsive design: Works on mobile 375px+
- Accessibility: Form labels, error messages, keyboard navigation (WCAG 2.1 AA)
- Consent checkbox: "Use my previous information" (allows patient to start fresh if preferred)

**Out of Scope:**
- Document upload (handled in EP-003)
- Real-time validation/field dependencies (keep form simple, Phase 1)
- Multi-language support (English only)
- Form versioning/branching (single form in Phase 1)

---

## Acceptance Criteria

1. **Form sections:** Form displays with tabs or accordion sections: "Chief Complaint", "Medical History", "Current Medications", "Allergies", "Insurance" (same as chatbot flow for consistency)
2. **Auto-population consent:** On form load, display: "We have your previous information. Use it to save time?" [Use Previous] [Start Fresh]
3. **Auto-population logic:** If patient selects [Use Previous], fetch last intake/visit data from database; pre-fill form fields with previous values
4. **Edit without friction:** All form fields are editable; no validation errors on load; patient can delete pre-filled values; can add new rows (e.g., new medication) without requiring entire form completion
5. **Chief complaint section:** Text field for chief complaint (why they're here); defaults to last visit's chief complaint if auto-populated; can be cleared/changed
6. **Medical history section:** Checkboxes for common conditions (diabetes, asthma, hypertension, etc.); also include text area for "other conditions"; auto-populated checkboxes from previous visit
7. **Medications section:** Repeating rows: [Medication Name] [Dosage] [Frequency] [Delete button]; auto-populated from previous visit; patient can add new rows by clicking "Add Medication"
8. **Allergies section:** Checkboxes/text for drug allergies; differentiates between "Allergies" (immune reaction) and "Side Effects" (GI upset); auto-populated; can add new
9. **Insurance section:** Fields: [Insurance Name] [Member ID] [Group Number] [Plan Name] (optional); auto-populated if available; same fields as chatbot collects
10. **Form submission:** "Submit" button at bottom (sticky on mobile); validation on submit only (not inline); shows clear error messages for required fields that are blank
11. **Timestamp preservation:** Form tracks last-modified timestamp for each section; displays "Last updated: [date]" in readonly text
12. **Mobile responsiveness:** Form is single-column on 375px; two-column on 768px+; "Add [item]" buttons are touch-friendly (44px+); sticky submit button on mobile

## Technical Notes

**Requirement Mapping:** FR-005 (Flexible Patient Intake - AI-Assisted & Manual Fallback)  
**Related Requirements:** NFR-005 (HIPAA), UI-001 (mobile-responsive), Web-Accessibility (WCAG 2.1 AA)  

**Technology Stack:**
- Frontend: React form component with React Hook Form or Formik for state management
- Repeating fields: react-hook-form fieldArray for medications/allergies
- Backend: `GET /api/patients/{patientId}/intake/last` → retrieves previous intake data
- Database: Query `IntakeResponses` or `VisitNotes` for most recent patient data

**Database Schema:**
```sql
CREATE TABLE IntakeResponses (
  id INT PRIMARY KEY,
  appointment_id INT FOREIGN KEY,
  patient_id INT FOREIGN KEY,
  mode VARCHAR ('ai', 'manual'),
  
  -- Chief Complaint
  chief_complaint VARCHAR (nullable),
  
  -- Medical History
  medical_history JSON, -- { conditions: ['diabetes', 'asthma'], other: 'string' }
  
  -- Medications
  medications JSON, -- [ { name: 'metformin', dosage: '500mg', frequency: '2x daily' }, ... ]
  
  -- Allergies
  allergies JSON, -- { drug_allergies: ['penicillin'], reactions: { penicillin: 'rash' }, side_effects: ['amoxicillin causes GI upset'] }
  
  -- Insurance
  insurance JSON, -- { name: 'Aetna', member_id: '12345', group_number: '67890', plan_name: 'PPO' }
  
  -- Metadata
  completed_at DATETIME (nullable),
  last_updated_at DATETIME,
  created_at DATETIME
);
```

**Auto-Population API:**
```
GET /api/patients/{patientId}/intake/last
Response: { 
  chief_complaint, 
  medical_history, 
  medications, 
  allergies, 
  insurance,
  last_updated_at 
}
```

**Form State Management Pattern:**
```
const [formData, setFormData] = useState({
  chief_complaint: previousIntake?.chief_complaint || '',
  medical_history: previousIntake?.medical_history || { conditions: [], other: '' },
  medications: previousIntake?.medications || [], // fieldArray
  allergies: previousIntake?.allergies || { drug_allergies: [], reactions: {} },
  insurance: previousIntake?.insurance || {}
});

const handleAddMedication = () => {
  setFormData(prev => ({
    ...prev,
    medications: [...prev.medications, { name: '', dosage: '', frequency: '' }]
  }));
};

const handleSubmitForm = async () => {
  // Validate required fields
  if (!formData.chief_complaint) {
    showError('Chief complaint is required');
    return;
  }
  // POST to /api/intake/submit
  await submitIntake(formData);
};
```

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Auto-populated data is outdated or incorrect → patient doesn't review → bad data | Medium | Always display "Last updated: [date]" prominently; encourage patient to review; add "Start Fresh" button as fallback |
| Form fields are tightly coupled to database schema → changing schema breaks form | Medium | Use schema migration strategy; version form structure; test form with old/new data combinations |
| Patient adds duplicate medications (e.g., adds "Metformin" when already pre-filled) → data inconsistency | Low | Implement medication name autocomplete with deduplication; warn if adding duplicate; allow delete before submit |
| Form validation too strict → patient can't proceed | Low | Only validate required fields on submit; make all fields optional except chief complaint; provide clear error messages |
| Mobile form is hard to fill (small inputs, hard-to-tap delete buttons) → abandonment | Medium | Test on real mobile devices; buttons minimum 44px; inputs minimum 16px font; test with VoiceOver/TalkBack |
