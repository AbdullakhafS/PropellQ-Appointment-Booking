# Task 002: Implement Intake Submission Persistence Service

**Task ID:** task_017_002  
**User Story:** US-017 (Store Intake Data in Structured Format)  
**Epic:** EP-002  
**Priority:** CRITICAL  
**Story Points:** 2  
**Owner Role:** Backend Developer  
**Status:** Not Started

## Objective

Implement backend persistence logic that writes intake payloads from chatbot or manual form into normalized tables in a single transactional flow.

## Requirements

1. Build persistence service for:
   - IntakeResponses master row
   - IntakeMedicalHistory entries
   - IntakeMedications entries
   - IntakeAllergies entries
   - IntakeInsurance record

2. Source support:
   - mode field supports ai and manual
   - confidence_score fields stored when provided by AI extraction

3. Transactional consistency:
   - All inserts for one intake request succeed or rollback together
   - No partial writes across detail tables

4. Metadata support:
   - completed_at, created_at, updated_at
   - created_by where available
   - notes field support

5. Compatibility:
   - Accept insurance verification status from US-015
   - Store values in IntakeInsurance consistently

## Implementation Steps

1. Implement repository/service layer methods per table.
2. Add transaction wrapper for full intake save.
3. Map request payload to normalized entities.
4. Handle optional sections gracefully.
5. Add unit tests for transactional behavior and mapping.

## Acceptance Criteria

- [ ] Full intake payload persists across all tables
- [ ] Rollback occurs on any insert failure
- [ ] Mode and confidence values saved correctly
- [ ] Insurance verification status stored correctly
- [ ] Service passes unit tests

## Dependencies

- Task 017_001 schema migration
- Intake payloads from US-011 and US-013
- Insurance status from US-015

## Validation

- [ ] Submit mixed payload with all sections and verify table contents
- [ ] Simulate failure in one detail table and confirm rollback
- [ ] Verify mode and confidence_score integrity

## Definition of Done

- Intake persistence service is transactional, stable, and production-ready.
