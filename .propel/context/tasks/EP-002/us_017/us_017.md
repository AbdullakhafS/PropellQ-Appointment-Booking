# US-017: Store Intake Data in Structured Format

**Epic:** EP-002 (Patient Intake & Insurance Verification)  
**Story Points:** 5  
**Priority:** HIGH  
**Status:** Not Started  
**Created:** 2026-06-17  

---

## User Story Statement

As a **system**, I want to **store all intake responses (from both chatbot and manual form) in a structured, normalized format that's queryable and linkable to clinical data**, so that **intake data can be aggregated into patient profiles and used for clinical decision-making and analytics**.

---

## Business Value

- **Outcome 1:** Structured storage enables intake data to flow into clinical profiles (EP-003)
- **Outcome 2:** Queryable format supports analytics (e.g., "Which allergies are most common?" or "No-show rate by insurance type?")
- **Outcome 3:** Normalized data reduces downstream processing errors and improves data quality

---

## Scope

**In Scope:**
- Define normalized schema for intake responses: chief complaint, medical history, medications, allergies, insurance
- Store intake responses with full traceability: timestamp, source (AI/manual), confidence scores
- Support for repeating fields (medications, allergies): one-to-many relationships
- Query optimization: Indexes for common queries (patient_id, appointment_id, created_at)
- Data versioning: Support for multiple intakes per patient (e.g., updated intake before visit)
- Immutability: Once stored, intake is never deleted (soft-delete if needed)

**Out of Scope:**
- Clinical interpretation of data (e.g., conflict detection) — handled in EP-003
- Auto-population of profiles from intake (handled in US-018)
- Analytics dashboards (Phase 2)

---

## Acceptance Criteria

1. **IntakeResponses table:** Master table storing one row per intake (per appointment)
   - Fields: id, appointment_id, patient_id, mode (ai/manual), completed_at, created_at, updated_at, notes
2. **Chief complaint storage:** Store chief_complaint text; linked to IntakeResponses
3. **Medical history storage:** Table `IntakeMedicalHistory` with one-to-many relationship to IntakeResponses
   - Fields: id, intake_id, condition_name, condition_code (ICD-10 if available), diagnosed_date (if known), status (active/inactive)
4. **Medications storage:** Table `IntakeMedications` with one-to-many relationship
   - Fields: id, intake_id, medication_name, dosage, frequency, route (oral/injection/etc), notes (if available)
5. **Allergies storage:** Table `IntakeAllergies` with one-to-many relationship
   - Fields: id, intake_id, allergen_type (drug/food/environmental), allergen_name, reaction_type (allergic/side_effect), reaction_description, severity (mild/moderate/severe)
6. **Insurance storage:** Table `IntakeInsurance` linked to IntakeResponses
   - Fields: id, intake_id, insurance_name, member_id, group_number, plan_name, verification_status
7. **Confidence scores:** Store with extracted data (relevant for AI-extracted data)
   - Field in each detail table: confidence_score (0-100)
8. **Audit trail:** `IntakeAuditLog` table tracks all changes to intake data
   - Fields: id, intake_id, changed_field, old_value, new_value, changed_by, changed_at
9. **Query performance:** Indexes on patient_id, appointment_id, created_at to support common queries
10. **Data validation on insert:** Validate data types, required fields, referential integrity
11. **Soft-delete support:** If intake needs to be invalidated, set status = 'voided' (never hard-delete)

## Technical Notes

**Requirement Mapping:** FR-005 (Flexible Patient Intake - AI-Assisted & Manual Fallback)  
**Related Requirements:** NFR-005 (HIPAA), NFR-004 (data quality), TR-003 (API standards), TR-006 (logging)  

**Technology Stack:**
- Database: PostgreSQL or SQL Server
- ORM: Entity Framework (.NET) or Hibernate (Java) for data access
- Validation: Data annotations + custom validators
- API: Repository pattern to encapsulate data access

**Database Schema:**
```sql
CREATE TABLE IntakeResponses (
  id INT PRIMARY KEY IDENTITY,
  appointment_id INT FOREIGN KEY NOT NULL,
  patient_id INT FOREIGN KEY NOT NULL,
  mode VARCHAR(20) NOT NULL DEFAULT 'manual', -- 'ai' or 'manual'
  completed_at DATETIME NOT NULL,
  status VARCHAR(20) NOT NULL DEFAULT 'completed', -- 'completed', 'voided'
  created_at DATETIME DEFAULT GETDATE(),
  updated_at DATETIME DEFAULT GETDATE(),
  created_by INT FOREIGN KEY (staff_id, nullable),
  notes VARCHAR(MAX) NULLABLE
);

CREATE TABLE IntakeMedicalHistory (
  id INT PRIMARY KEY IDENTITY,
  intake_id INT FOREIGN KEY NOT NULL,
  condition_name VARCHAR(255) NOT NULL, -- 'diabetes', 'asthma', etc.
  condition_code VARCHAR(10) NULLABLE, -- ICD-10 code if available
  diagnosed_date DATE NULLABLE,
  status VARCHAR(20) NOT NULL DEFAULT 'active', -- 'active', 'inactive', 'resolved'
  confidence_score INT DEFAULT 100, -- 0-100, relevant for AI-extracted
  notes VARCHAR(500) NULLABLE,
  created_at DATETIME DEFAULT GETDATE()
);

CREATE TABLE IntakeMedications (
  id INT PRIMARY KEY IDENTITY,
  intake_id INT FOREIGN KEY NOT NULL,
  medication_name VARCHAR(255) NOT NULL,
  dosage VARCHAR(100) NULLABLE, -- '500mg', '10mg/mL', etc.
  frequency VARCHAR(100) NULLABLE, -- '2x daily', 'every 8 hours', etc.
  route VARCHAR(50) NULLABLE, -- 'oral', 'injection', 'inhaler', etc.
  confidence_score INT DEFAULT 100,
  notes VARCHAR(500) NULLABLE,
  created_at DATETIME DEFAULT GETDATE()
);

CREATE TABLE IntakeAllergies (
  id INT PRIMARY KEY IDENTITY,
  intake_id INT FOREIGN KEY NOT NULL,
  allergen_type VARCHAR(50) NOT NULL, -- 'drug', 'food', 'environmental'
  allergen_name VARCHAR(255) NOT NULL,
  reaction_type VARCHAR(50) NOT NULL, -- 'allergic', 'side_effect'
  reaction_description VARCHAR(500),
  severity VARCHAR(50) NULLABLE, -- 'mild', 'moderate', 'severe'
  confidence_score INT DEFAULT 100,
  created_at DATETIME DEFAULT GETDATE()
);

CREATE TABLE IntakeInsurance (
  id INT PRIMARY KEY IDENTITY,
  intake_id INT FOREIGN KEY NOT NULL,
  insurance_name VARCHAR(255),
  member_id VARCHAR(255),
  group_number VARCHAR(255) NULLABLE,
  plan_name VARCHAR(255) NULLABLE,
  verification_status VARCHAR(50), -- 'verified', 'unverified', 'manual_review'
  created_at DATETIME DEFAULT GETDATE()
);

CREATE TABLE IntakeAuditLog (
  id INT PRIMARY KEY IDENTITY,
  intake_id INT FOREIGN KEY,
  changed_field VARCHAR(255),
  old_value VARCHAR(MAX),
  new_value VARCHAR(MAX),
  changed_by INT FOREIGN KEY (staff_id, nullable),
  changed_at DATETIME DEFAULT GETDATE(),
  reason VARCHAR(500) NULLABLE
);

-- Indexes for query performance
CREATE INDEX idx_intake_patient ON IntakeResponses(patient_id);
CREATE INDEX idx_intake_appointment ON IntakeResponses(appointment_id);
CREATE INDEX idx_intake_created ON IntakeResponses(created_at DESC);
CREATE INDEX idx_medical_history_intake ON IntakeMedicalHistory(intake_id);
CREATE INDEX idx_medications_intake ON IntakeMedications(intake_id);
CREATE INDEX idx_allergies_intake ON IntakeAllergies(intake_id);
CREATE INDEX idx_insurance_intake ON IntakeInsurance(intake_id);
```

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Unstructured/malformed data is stored → breaks downstream processing | Medium | Validate all inputs before insert; test with edge cases; add data quality checks |
| Storage grows unbounded → database bloat and slow queries | Low | Archive old intakes (>1 year) to separate schema; implement data retention policy |
| Confidential fields (SSN, financial data) accidentally stored → HIPAA violation | Critical | Implement field-level PII detection; never accept SSN or full financial data; mask PII in audit logs |
| Concurrent intake updates (user edits same intake) → data conflict | Low | Implement optimistic locking (version column); last-write-wins or explicit conflict resolution |
