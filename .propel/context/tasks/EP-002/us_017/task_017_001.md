# Task 001: Design and Create Normalized Intake Schema

**Task ID:** task_017_001  
**User Story:** US-017 (Store Intake Data in Structured Format)  
**Epic:** EP-002  
**Priority:** CRITICAL  
**Story Points:** 2  
**Owner Role:** Backend Developer / DBA  
**Status:** Done

## Objective

Create the normalized database schema for intake data across master and detail tables with referential integrity and soft-delete support.

## Requirements

1. Create tables:
   - IntakeResponses
   - IntakeMedicalHistory
   - IntakeMedications
   - IntakeAllergies
   - IntakeInsurance
   - IntakeAuditLog

2. Relationship rules:
   - One intake response per appointment submission
   - One-to-many links from IntakeResponses to medical history, medications, and allergies
   - One-to-one or one-to-many configurable for insurance details linked by intake_id

3. Required fields and constraints:
   - Required: appointment_id, patient_id, mode, completed_at
   - Status enum for IntakeResponses: completed, voided
   - Foreign keys and cascade behavior defined explicitly

4. Soft-delete support:
   - No hard deletion of intake records
   - Use status = voided for invalidation

5. Indexing:
   - IntakeResponses: patient_id, appointment_id, created_at
   - Detail tables: intake_id indexes

## Implementation Steps

1. Draft schema migration scripts.
2. Define FK and status constraints.
3. Add indexes for common query paths.
4. Add schema validation checks and migration rollback support.
5. Conduct DBA review.

## Acceptance Criteria

- [x] All required tables created successfully
- [x] Relationships and constraints enforced
- [x] Status-based soft-delete supported
- [x] Indexes exist for required query fields
- [x] Migration runs cleanly in test and staging

## Dependencies

- Existing appointment and patient tables
- DB migration framework

## Validation

- [x] Insert valid intake across all detail tables succeeds
- [x] Invalid FK insert is rejected
- [x] Voided status is persisted correctly

## Definition of Done

- Normalized intake schema is implemented, constrained, and ready for API integration.
