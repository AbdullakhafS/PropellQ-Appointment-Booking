# Task 004: Implement Intake Audit Logging and Change Tracking

**Task ID:** task_017_004  
**User Story:** US-017 (Store Intake Data in Structured Format)  
**Epic:** EP-002  
**Priority:** HIGH  
**Story Points:** 1  
**Owner Role:** Backend Developer  
**Status:** Not Started

## Objective

Capture an immutable audit trail for intake inserts and updates so every data change is traceable by field, actor, and timestamp.

## Requirements

1. Audit table usage:
   - Write IntakeAuditLog entries for create/update/void actions
   - Store changed_field, old_value, new_value, changed_by, changed_at, reason

2. Logging behavior:
   - Log all updates to detail tables and status changes in IntakeResponses
   - Ensure no PII overexposure in log values where masking is needed

3. Consistency:
   - Audit log writes should occur within the same transaction context as data changes when possible

4. Queryability:
   - Support retrieval of change timeline by intake_id

## Implementation Steps

1. Implement audit writer utility/service.
2. Hook audit logging into persistence and update flows.
3. Add masking rules for sensitive values if necessary.
4. Create tests for audit row correctness.

## Acceptance Criteria

- [ ] Create/update/void actions generate audit log rows
- [ ] Audit rows include actor, timestamp, old/new values
- [ ] Intake change timeline is queryable by intake_id
- [ ] Audit tests pass

## Dependencies

- Task 017_001 schema including IntakeAuditLog
- Persistence flows from task_017_002

## Validation

- [ ] Update intake section and verify accurate audit row
- [ ] Void intake and verify status change log
- [ ] Audit log query returns chronological timeline

## Definition of Done

- Intake audit logging is complete, accurate, and compliant.
