# Task 004: Build Intake Update API with Authorization and Audit Tracking

**Task ID:** task_018_004  
**User Story:** US-018 (Display Intake Responses in Patient Profile)  
**Epic:** EP-002  
**Priority:** HIGH  
**Story Points:** 1  
**Owner Role:** Backend Developer  
**Status:** Not Started

## Objective

Implement PATCH endpoint for profile-based intake updates with strict authorization, validation, and audit logging.

## Requirements

1. Endpoint:
   - PATCH /api/patients/{patientId}/intake/{intakeId}
2. Validation:
   - Accept updates to allowable intake fields only
   - Validate nested section types and required constraints
3. Authorization:
   - Ensure requester can edit this patient intake
   - Deny edits if appointment is already started/checked_in
4. Persistence:
   - Update normalized intake records from US-017
   - Update IntakeResponses.updated_at
5. Audit logging:
   - Write IntakeAuditLog entries for changed fields and values

## Implementation Steps

1. Define editable DTO and field whitelist.
2. Implement appointment state validation for edit lock.
3. Implement patch mapping to normalized tables.
4. Write audit records per modified field.
5. Add integration tests for success and rejection cases.

## Acceptance Criteria

- [ ] PATCH endpoint updates intake data correctly
- [ ] Unauthorized or locked appointments cannot be edited
- [ ] updated_at reflects successful edits
- [ ] Audit log captures changes
- [ ] Integration tests pass

## Dependencies

- US-017 persistence and audit schema
- Appointment state/status source

## Validation

- [ ] Editable intake updates persist across all relevant tables
- [ ] checked_in appointment edit returns proper error
- [ ] Audit entries reflect actual changes

## Definition of Done

- Intake update API is secure, validated, and auditable.
