# Task 001: Build Latest Intake Retrieval API for Patient Profile

**Task ID:** task_018_001  
**User Story:** US-018 (Display Intake Responses in Patient Profile)  
**Epic:** EP-002  
**Priority:** CRITICAL  
**Story Points:** 2  
**Owner Role:** Backend Developer  
**Status:** Done

## Objective

Implement an API endpoint to retrieve the most recent completed intake for a patient with all normalized sections combined into a profile-friendly response.

## Requirements

1. Endpoint:
   - GET /api/patients/{patientId}/intake/latest
2. Return sections:
   - chief complaint, medical history, medications, allergies, insurance
   - metadata: completed_at, mode, version, last_updated_at
3. Query logic:
   - Fetch latest completed intake by created_at DESC (or appointment filter)
   - Join normalized tables from US-017
4. Security:
   - Ensure only authorized patient or staff can access record
5. Error handling:
   - 404 when no intake exists
   - 403 for unauthorized access

## Implementation Steps

1. Define response DTO for profile intake payload.
2. Implement latest-intake query with required joins.
3. Add authorization checks for patient/staff role access.
4. Add endpoint/controller and integration tests.
5. Optimize query path for profile load performance.

## Acceptance Criteria

- [x] Endpoint returns latest intake with all sections and metadata
- [x] Access is role- and ownership-protected
- [x] No-intake case handled gracefully
- [x] Integration tests pass
- [x] Response latency is suitable for profile page load (<500ms)

## Dependencies

- US-017 normalized intake schema
- Profile auth context

## Validation

- [x] Latest intake returned correctly when multiple intakes exist
- [x] Unauthorized user receives 403
- [x] Missing intake returns expected response

## Definition of Done

- Latest intake API is stable, secure, and ready for profile integration.
