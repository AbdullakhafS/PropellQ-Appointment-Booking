# Task 005: Integrate Mode Switching with Chatbot and Form Workflows

**Task ID:** task_014_005  
**User Story:** US-014 (Enable Intake Mode Switching)  
**Epic:** EP-002  
**Priority:** HIGH  
**Story Points:** 1  
**Owner Role:** Frontend/Backend Developer  
**Status:** Not Started

## Objective

Connect the mode switching logic to the actual chatbot and form workflows so the user can switch modes mid-intake without losing data.

## Requirements

1. Trigger mode switch from the intake UI.
2. On chatbot → form switch:
   - Map chatbot data into the manual form structure
   - Load the form with pre-filled values
   - Preserve conversation progress in a draft

3. On form → chatbot switch:
   - Convert form values into chatbot summary payloads
   - Initiate chatbot conversation from the current state
   - Preserve form progress in a draft

4. Switch state handling:
   - Update `mode` and `switchCount`
   - Set `switched_to_manual_at` or similar metadata
   - Render the correct UI for the new mode

5. Error handling:
   - If mapping fails, show a graceful error and allow user to retry
   - If draft save fails, warn the user but do not lose current data

## Implementation Steps

1. Wire switch actions to mapping functions and state updates.
2. Add draft save calls on switch.
3. Ensure the new mode loads the correct payload.
4. Show progress and preserve previous entries.
5. Add tests for the switch integration.

## Acceptance Criteria

- [ ] Mode switch during intake transitions to the other mode
- [ ] Data from the original mode appears correctly in the target mode
- [ ] Draft state persists across the switch
- [ ] Switch count increments and is enforced
- [ ] Errors are handled without losing data

## Dependencies

- Task 014_001 mode state and persistence
- Task 014_002 data mapping
- Task 014_003 switch UI
- Task 014_004 draft API

## Validation

- [ ] Chatbot → form switch preserves data accurately
- [ ] Form → chatbot switch preserves data accurately
- [ ] Draft persists after switching and reload
- [ ] No mode switch loops occur beyond the allowed limit

## Definition of Done

- Mode switching integrated end-to-end with both intake workflows.
