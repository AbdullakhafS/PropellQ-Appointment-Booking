# US-014: Enable Intake Mode Switching (AI ↔ Manual)

**Epic:** EP-002 (Patient Intake & Insurance Verification)  
**Story Points:** 5  
**Priority:** HIGH  
**Status:** Not Started  
**Created:** 2026-06-17  

---

## User Story Statement

As a **patient**, I want to **seamlessly switch from AI chatbot intake to manual form (or vice versa) at any point without losing my previously entered data**, so that **I can choose the intake method that works best for me, and I'm not frustrated if one method doesn't feel right**.

---

## Business Value

- **Outcome 1:** Patient choice increases intake completion rate (≥85%); some patients prefer forms, others prefer conversational AI
- **Outcome 2:** If chatbot is confusing, patient can switch to form without restarting; reduces abandonment
- **Outcome 3:** If form is tedious, patient can switch to chatbot mid-way; flexibility increases satisfaction

---

## Scope

**In Scope:**
- UI toggle/button: "Switch to [other mode]" visible at any time during intake
- Data preservation: Switching preserves all previously entered data; no loss
- Mode state tracking: Track which mode patient started in; allow switching once per intake session
- Seamless transition: Switching takes <1 second; no page reload if possible
- Data mapping: Map chatbot-extracted data → manual form fields (and vice versa) so patient sees their data in new format
- Progress preservation: Progress indicator updates (e.g., "You're 50% done" stays consistent across modes)

**Out of Scope:**
- Allow unlimited switching (one switch per intake session to avoid confusion)
- Real-time bi-directional sync (once data is entered in one mode, switching shows that data; no constant sync)
- Complex branching based on mode switch (simple UI state change)

---

## Acceptance Criteria

1. **Mode selection UI:** After patient completes any section (chief complaint, medical history, etc.), display "Or switch to [Form / Chatbot]?" button/link prominently
2. **Initial mode choice:** Patient first chooses mode at intake start: "How would you prefer to complete intake?" [Use AI Assistant] [Fill Out Form] (if neither selected, default to AI)
3. **Switching button visible:** Throughout intake, display a "Switch to [other mode]" button (not hidden in menu); easily accessible on mobile and desktop
4. **Data preservation on switch:** When patient clicks "Switch to Form" (from chatbot), system:
   - Retrieves all chatbot responses entered so far
   - Maps chatbot-extracted data → form fields (e.g., medications extracted by chatbot populate medication repeating fields in form)
   - Displays form with pre-filled data from chatbot conversation
   - Shows indication: "You've already answered these questions in the chatbot. Review and edit as needed."
5. **Reverse switching:** When switching from form → chatbot:
   - Retrieve form field values
   - Convert form data → natural language prompts (e.g., "I see you entered metformin 500mg twice daily. Is that correct?" instead of re-asking)
   - Continue chatbot conversation from that point
6. **Data mapping logic:** 
   - Chatbot medications [{ name: 'metformin', dosage: '500mg', frequency: '2x daily' }] → Form medications repeating rows with same values pre-filled
   - Chatbot allergies { drug_allergies: ['penicillin'] } → Form allergy checkboxes + text fields pre-checked
   - Form values → Chatbot conversation: "You've told me you have diabetes. Anything else?"
7. **Switch limit:** Only allow one mode switch per intake session (to avoid infinite switching/confusion); after switching once, disable further switches (or allow 2 switches max)
8. **Progress indicator:** After switching, progress indicator updates (e.g., "Step 3 of 6 in Form mode" or "Question 5 of 9 in Chatbot mode"); shows patient where they are in new mode
9. **Mobile UX:** Switch button is clearly visible (not hidden in hamburger menu); takes up ~10% of screen real estate
10. **Confirmation on switch:** Display confirmation modal: "Switch to [other mode]? Your responses will be preserved." [Yes, Switch] [Cancel]

## Technical Notes

**Requirement Mapping:** FR-005 (Flexible Patient Intake - AI-Assisted & Manual Fallback)  
**Related Requirements:** NFR-005 (HIPAA), UI-001 (responsive UX)  

**Technology Stack:**
- Frontend: React context/Redux to manage intake mode state
- Data mapping: Transformation functions convert between chatbot output format ↔ form field format
- Session storage: Use localStorage to persist mode state + partial data during session

**Database & Session State:**
```javascript
// Frontend state
const [intakeState, setIntakeState] = useState({
  mode: 'ai' | 'manual',
  hasInitialized: false,
  data: {
    chief_complaint: '',
    medical_history: {},
    medications: [],
    allergies: {},
    insurance: {}
  },
  switchCount: 0,
  lastSwitchedAt: null
});

// Backend API to save/retrieve partial intake state
POST /api/intake/{appointmentId}/save-draft
  { mode, data, switchCount }
  
GET /api/intake/{appointmentId}/draft
  → { mode, data, switchCount, lastUpdated }
```

**Data Mapping Functions:**

```typescript
// Convert chatbot output → form fields
function mapChatbotToForm(chatbotData) {
  return {
    chief_complaint: chatbotData.chief_complaint,
    medical_history: chatbotData.medical_history,
    medications: chatbotData.medications.map(m => ({
      name: m.name,
      dosage: m.dosage,
      frequency: m.frequency
    })),
    allergies: {
      drug_allergies: chatbotData.allergies.drug_allergies,
      reactions: chatbotData.allergies.reactions
    },
    insurance: chatbotData.insurance
  };
}

// Convert form fields → chatbot conversation
function mapFormToChatbot(formData) {
  // Convert form structure back to natural language responses
  // E.g., medications array → "I'm taking [med1] [dosage], [med2] [dosage], ..."
  const medicationsSummary = formData.medications
    .map(m => `${m.name} ${m.dosage}`)
    .join(', ');
  
  return {
    chief_complaint: formData.chief_complaint,
    medications_summary: medicationsSummary,
    allergies_summary: formData.allergies.drug_allergies.join(', '),
    // ... etc
  };
}
```

**Switch Flow (UI/UX):**
```
User in Chatbot Mode
  → Click "Switch to Form"
  → Modal: "Switch to Form? Your responses will be preserved."
  → [Yes, Switch] [Cancel]
  
  (if Yes, Switch):
    → API: POST /api/intake/draft { mode: 'form', data: {...} }
    → mapChatbotToForm(chatbotData)
    → Render Form with pre-populated data
    → Disable "Switch to Chatbot" button (only 1 switch allowed)
```

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Data mapping is incomplete → switching loses some fields | High | Unit test all mapping functions; test with diverse input data; validate mapping round-trip (A→B→A) |
| Patient switches back-and-forth multiple times → confusion about data consistency | Medium | Limit switches to 2 max; display warning: "You're switching modes; make sure your responses are consistent." |
| Switch introduces latency (data transformation takes time) → poor UX | Low | Optimize mapping functions; test performance; use memoization to cache transformations |
| Mobile switch button obscures form fields → bad UX | Low | Position button at top or bottom (sticky); test on real mobile devices; ensure button doesn't overlap form |
