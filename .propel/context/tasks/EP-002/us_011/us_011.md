# US-011: Implement AI Chatbot Engine (NLP Integration)

**Epic:** EP-002 (Patient Intake & Insurance Verification)  
**Story Points:** 8  
**Priority:** HIGH  
**Status:** Not Started  
**Created:** 2026-06-17  

---

## User Story Statement

As a **patient**, I want to **interact with an AI-powered chatbot that understands natural language and guides me through medical history, medications, allergies, and chief complaint**, so that **intake feels conversational rather than like filling out a form, increasing completion rate and data quality**.

---

## Business Value

- **Outcome 1:** AI-assisted intake adoption ≥50%; differentiates platform vs. traditional form-based intake
- **Outcome 2:** Reduces intake time from 20 minutes (manual form) to 5-7 minutes (conversational AI)
- **Outcome 3:** Higher data quality; conversational AI captures nuance (e.g., "allergic to penicillin rash" vs. just "penicillin")

---

## Scope

**In Scope:**
- Integrate NLP chatbot engine (OpenAI GPT, LLaMA, or similar)
- Chatbot supports conversational Q&A for: chief complaint, medical history, medication list, allergies, insurance info
- Natural language understanding: Patient can respond naturally ("I'm allergic to penicillin" or "Penicillin makes me break out")
- Conversation context tracking: Chatbot remembers previous answers; asks follow-up questions intelligently
- Fallback for misunderstandings: If chatbot can't parse response, ask for clarification without being frustrating
- Conversation transcript logging: Store all Q&A pairs for audit trail and future ML training
- Supports 1-3 minute response time per turn (real-time interactive feel)

**Out of Scope:**
- Speech-to-text or voice input (text-only in Phase 1)
- Multi-language support (English only in Phase 1)
- Advanced NER (named entity recognition) for drug names (basic matching in Phase 1)
- Multimodal input (documents, images) — handled separately in EP-003

---

## Acceptance Criteria

1. **Chatbot interface loads:** Patient clicks "Start AI Intake" or selects AI mode during pre-visit; chatbot UI appears in modal or dedicated page with conversation history visible
2. **Initial greeting:** Chatbot greets patient by name; explains intake will take ~5-7 minutes; asks first question: "What brings you in today?" (chief complaint)
3. **Chief complaint capture:** Patient types natural language response (e.g., "I have a cough for 3 days"); chatbot parses and stores response; asks follow-up: "How long has the cough been going on?" or "Any chest pain?"
4. **Medical history collection:** Chatbot asks about relevant medical conditions: "Do you have any chronic conditions like diabetes, asthma, or heart disease?" Patient responds naturally; chatbot classifies and stores
5. **Medication list extraction:** Chatbot asks: "What medications are you currently taking?" Patient responds (e.g., "I take metformin and lisinopril"); chatbot parses medication names and dosages from natural text
6. **Allergy detection:** Chatbot asks: "Do you have any drug allergies?" Captures responses; differentiates between allergies (immune reaction) and side effects (e.g., "penicillin causes rash" vs. "amoxicillin upsets my stomach")
7. **Conversation context:** If patient mentions "allergic to penicillin" early on, and later mentions "amoxicillin," chatbot recognizes amoxicillin is a penicillin derivative and asks: "You mentioned penicillin allergy; are you also allergic to amoxicillin?"
8. **Misunderstanding handling:** If patient says something chatbot can't parse (e.g., nonsensical or off-topic), chatbot responds politely: "I'm not sure I understood. Could you rephrase that?" Maximum 2 re-attempts before asking patient to skip or switch to manual form
9. **Completion summary:** After all questions answered, chatbot displays summary: "Here's what I captured: [chief complaint], [medical history], [medications], [allergies]" with option to edit, add missing info, or confirm
10. **Response time:** Each chatbot turn completes within 3 seconds (includes API latency, NLP inference, response generation)

## Technical Notes

**Requirement Mapping:** FR-005 (Flexible Patient Intake - AI-Assisted & Manual Fallback)  
**Related Requirements:** NFR-005 (HIPAA data handling), NFR-007 (logging), TR-003 (API standards)  

**Technology Stack:**
- NLP Engine: OpenAI GPT-3.5/4, Azure OpenAI, or open-source LLaMA via inference service
- Backend: .NET/Java API wrapping NLP model with prompt engineering
- Frontend: React chatbot component (e.g., react-chat-elements, custom)
- Conversation Storage: `IntakeConversations` table storing Q&A pairs

**Database Schema:**
```sql
CREATE TABLE IntakeConversations (
  id INT PRIMARY KEY,
  appointment_id INT FOREIGN KEY,
  patient_id INT FOREIGN KEY,
  mode VARCHAR ('ai', 'manual'),
  conversation_transcript JSON, -- Array of { role: 'user'|'assistant', content: string, timestamp }
  extracted_data JSON, -- { chief_complaint, medical_history, medications, allergies }
  confidence_scores JSON, -- { chief_complaint: 0.95, meds: 0.88, ... }
  completed_at DATETIME (nullable),
  switched_to_manual_at DATETIME (nullable),
  created_at DATETIME
);

CREATE TABLE ChatbotPrompts (
  id INT PRIMARY KEY,
  prompt_version VARCHAR (e.g., '1.0'),
  prompt_text VARCHAR (system prompt for chatbot),
  effective_date DATETIME,
  deprecated_at DATETIME (nullable)
);
```

**Chatbot Prompt Template (System Prompt):**
```
You are a medical intake assistant for a healthcare platform. 
Your role is to:
1. Ask the patient about their chief complaint (reason for visit)
2. Gather relevant medical history
3. List current medications with dosages
4. Identify drug allergies and reactions
5. Verify insurance information (name, ID, group number)

Keep responses conversational and empathetic. After each response, ask ONE follow-up question.
When the patient provides information, extract structured data in the background.
If the patient's response is unclear, ask for clarification in a friendly way.

Do NOT:
- Provide medical advice or diagnosis
- Recommend treatments
- Share concerns about patient data beyond what they've told you

After gathering all required information, summarize what you've learned and ask for confirmation.
```

**NLP Integration Workflow:**
```
1. User sends message in chatbot UI
2. Frontend POST /api/intake/chat { appointmentId, message, conversationId }
3. Backend retrieves previous conversation context from DB
4. Constructs LLM prompt with conversation history + system prompt
5. Calls OpenAI/LLaMA API with prompt
6. Receives response from LLM
7. Extracts structured data from response (NER, classification)
8. Stores Q&A pair + extracted data in IntakeConversations
9. Returns response to frontend for display
10. Frontend displays chatbot message + awaits user input
```

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| NLP engine misunderstands patient input → incorrect data extraction | High | Use low-confidence threshold (e.g., <70%) as trigger for manual review; implement human-in-the-loop verification; test with diverse patient responses in QA |
| API latency from NLP service exceeds 3 seconds → poor UX | Medium | Implement response caching for common questions; use faster inference model or local inference if available; set timeout and fallback to manual form |
| Chatbot reveals sensitive data (SSN, account number) in conversation → HIPAA violation | Critical | Scrub sensitive PII from conversation logs; use content filtering; don't ask for SSN or full account numbers (only last 4 digits) |
| Patient abandons chatbot mid-conversation → incomplete data | Medium | Auto-save conversation progress; allow patient to resume; implement exit flow that saves partial data |
| NLP training data bias → chatbot performs poorly for non-English speakers or minority groups | Medium | Test with diverse patient cohorts; document limitations; provide manual form as alternative |
