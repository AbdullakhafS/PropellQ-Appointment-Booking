# US-104: Finalize Production Schema and Index Strategy

**Story ID:** US-104  
**Epic:** EP-DATA-001 (Data Platform & Governance)  
**Priority:** CRITICAL  
**Story Points:** 8  
**Owner Role:** Data Architect / Backend Engineer  
**Status:** Draft

## User Story

As a data architect, I want a production-ready relational schema and index strategy so that booking, profile, and clinical queries remain correct and fast at scale.

## Business Value

- Prevents data model drift and inconsistent entity semantics.
- Improves response times for high-frequency operational workflows.
- Reduces long-term maintenance cost by defining strong schema governance up front.

## Scope

In scope:
- Canonical schema for appointments, patient profile, intake, documents, coding, and audit references.
- Primary/foreign keys, uniqueness constraints, and check constraints.
- Index strategy for hot read/write patterns (booking lookup, queue, profile timeline).
- Query plan baseline for critical endpoints.

Out of scope:
- Full historical data backfill from external legacy systems.

## Acceptance Criteria

1. Given the production schema definition, when reviewed, then all core entities include explicit PK/FK constraints and documented cardinality.
2. Given critical operational queries, when benchmarked on representative data volume, then p95 query latency meets the agreed budget.
3. Given duplicate-risk entities (appointments, patient identifiers), when inserts/updates occur, then uniqueness constraints prevent duplicate records.
4. Given schema artifacts, when consumed by service teams, then naming conventions and column semantics are consistent and documented.
5. Given index candidates, when query plans are analyzed, then redundant or unused indexes are removed and hot-path indexes are retained.
6. Given schema changes are proposed, when submitted, then they pass architecture review with compatibility notes.

## Technical Notes

- Target requirements: DR-001, DR-002, NFR-004.
- Include read/write path matrix for booking, intake, clinical aggregation, and dashboard reads.
- Include partitioning guidance if dataset growth thresholds are crossed.

## Dependencies

- API access patterns from EP-TECH-001 standards.
- Input from EP-001, EP-003, EP-006 for hot query paths.

## Risks and Mitigations

- Risk: Over-indexing degrades write performance.
- Mitigation: Validate with mixed workload benchmark and monitor index usage.

## Definition of Done

- Approved production schema and DDL set committed.
- Index strategy documented with benchmark evidence.
- Query plan snapshots stored for critical endpoints.
- Data model glossary updated.
