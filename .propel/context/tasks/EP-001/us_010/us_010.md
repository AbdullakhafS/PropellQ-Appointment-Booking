# US-010: Mobile-Responsive Booking Experience

**Epic:** EP-001 (Patient Appointment Booking)  
**Story Points:** 5  
**Priority:** CRITICAL  
**Status:** Not Started  
**Created:** 2026-06-17  

---

## User Story Statement

As a **patient**, I want to **book an appointment seamlessly on my mobile phone (375px width) with a responsive design that adapts to tablet (768px) and desktop (1024px+) screens**, so that **I can book an appointment anytime, anywhere, without friction from poor mobile UX**.

---

## Business Value

- **Outcome 1:** Mobile-first design drives higher booking completion; 60%+ of users book on mobile
- **Outcome 2:** Responsive experience across devices increases accessibility
- **Outcome 3:** Supports business goal of self-service bookings (reduces front-desk calls)

---

## Scope

**In Scope:**
- Responsive design for three breakpoints: 375px (mobile), 768px (tablet), 1024px+ (desktop)
- Touch-friendly interactions: buttons 44px+ tall, tap targets spaced 8px apart
- Fast mobile performance: page load <3 seconds on 4G, LCP <2.5s
- Mobile navigation: hamburger menu or bottom navigation for main sections
- Touch gestures: swipe for calendar month navigation, tap-to-select appointments
- Keyboard accessibility: fully navigable via keyboard on all devices
- Dark mode support (nice-to-have): system prefers-color-scheme support

**Out of Scope:**
- Native mobile app (web-only in Phase 1)
- Offline booking (requires service workers, Phase 2)
- Voice/gesture-based booking (Phase 2)

---

## Acceptance Criteria

1. **Mobile breakpoint (375px):** 
   - Search filters stack vertically; dropdown menus expand on tap
   - Calendar displays as scrollable month grid or single-week view
   - Appointment details displayed in card format (not collapsed modals)
   - Checkout form is single-column, no side-by-side fields
   - "Book Now" button spans full width, 48px height, easy to tap

2. **Tablet breakpoint (768px):**
   - Search filters display in two-column layout or sidebar
   - Calendar shows 2-week view by default
   - Checkout form has 2-column layout if space permits
   - Navigation accessible via bottom tab bar or top nav

3. **Desktop breakpoint (1024px+):**
   - Search filters in left sidebar (sticky)
   - Calendar shows full month view
   - Checkout form displays with details sidebar
   - Full navigation with dropdown menus

4. **Performance metrics:**
   - First Contentful Paint (FCP): <1.5s on 4G network
   - Largest Contentful Paint (LCP): <2.5s on 4G network
   - Cumulative Layout Shift (CLS): <0.1 (no unexpected layout shifts)
   - Mobile lighthouse score: ≥80 (performance)

5. **Touch interactions:**
   - All buttons minimum 44px x 44px (iOS) or 48px x 48px (Android)
   - Tap targets spaced minimum 8px apart
   - No hover-dependent interactions (fallback for touch)
   - Swipe gestures for calendar navigation (left/right swipe = prev/next month)

6. **Accessibility on mobile:**
   - Font size minimum 16px (prevents iOS zoom on input focus)
   - Color contrast ≥4.5:1 on text
   - All form labels linked to inputs
   - Error messages associated with inputs (aria-describedby)
   - Screen reader support: all interactive elements have accessible names

7. **Image optimization:**
   - Provider photos compressed and responsive (max 200KB per image)
   - SVG icons instead of raster icons where possible
   - Lazy-load images below fold

8. **Form optimization for mobile:**
   - Use native input types: email, tel, date (invokes native keyboards)
   - Single form per page (no multi-page checkout on mobile)
   - Autocomplete attributes enabled (name, email, tel, address)
   - Submit button visible without scrolling (sticky footer on mobile)

## Technical Notes

**Requirement Mapping:** FR-001 (Patient Self-Service Appointment Booking)  
**Related Requirements:** NFR-002 (sub-500ms API latency), UI-001 (mobile-responsive design), Web-Accessibility (WCAG 2.1 AA)  

**Technology Stack:**
- CSS Framework: CSS Grid + Flexbox (no framework dependency)
- Mobile-first approach: Base styles for 375px, media queries for larger breakpoints
- Performance: Code-split components; lazy-load heavy libraries (calendar, charts)
- Testing: Responsive design testing tools (Chrome DevTools, BrowserStack)

**Breakpoint Configuration:**
```css
/* Mobile-first */
@media (min-width: 768px) { /* Tablet */ }
@media (min-width: 1024px) { /* Desktop */ }
```

**Critical Rendering Path Optimization:**
- Inline critical CSS (above-the-fold)
- Defer non-critical CSS loading
- Defer JavaScript unless needed for interactivity
- Load fonts asynchronously (swap strategy)
- Preload high-priority resources (appointment search API)

**Mobile Navigation Pattern:**
```
Mobile (375px):
  Top: Logo + hamburger menu
  Content: Full-width cards, single column
  Bottom: Sticky action buttons ("Next", "Book Now")
  Menu: Slides in from left/right

Tablet (768px):
  Top: Logo + navigation tabs
  Sidebar: Filters (sticky, collapse on small tablets)
  Content: Two-column or two-row layout

Desktop (1024px+):
  Top: Full navigation bar
  Left Sidebar: Filters and context
  Main: Calendar + details
  Right Sidebar: Summary/confirmation
```

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Performance on low-end mobile devices (3G, older phones) → slow load times | High | Implement aggressive code-splitting; measure performance on Moto G4 / Galaxy A10; set up continuous performance monitoring |
| iOS Safari font size auto-zoom on input focus → UX disruption | Medium | Set font-size ≥16px on inputs; test on real iPhone; document for all input components |
| Calendar is slow on mobile with many slots | Medium | Virtualize slot rendering; show only visible slots in viewport; lazy-load month data on scroll |
| Touch target sizes conflict with design aesthetic → buttons appear large on desktop | Low | Use media queries to vary button padding; desktop buttons can be smaller (CSS 2-column layout) |
| Responsive image srcset not working on older browsers → large images downloaded | Low | Provide fallback img tag; test on IE11 (Phase 1 requirement?); document browser support |

- [ ] Lighthouse audit passed (≥80 on performance, ≥90 on accessibility)

- [ ] Font loading optimized (avoid invisible text, font-display: swap)

- [ ] Accessibility tested: keyboard navigation, screen reader (VoiceOver, NVDA)

- [ ] E2E tests: Mobile booking flow (search → book) on simulated 4G
