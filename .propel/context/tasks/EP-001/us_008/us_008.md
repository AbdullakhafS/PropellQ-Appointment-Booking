# US-008: Authorize Outlook Integration

**Epic:** EP-001 (Patient Appointment Booking)  
**Story Points:** 5  
**Priority:** HIGH  
**Status:** Not Started  
**Created:** 2026-06-17  

---

## User Story Statement

As a **patient**, I want to **authorize the PropellQ application to access my Outlook (Microsoft Graph) calendar**, so that **my appointments are automatically added to my Outlook calendar and I can manage them alongside my work meetings**.

---

## Business Value

- **Outcome 1:** Supports patients who use Outlook/Office 365; increases addressable market
- **Outcome 2:** Same benefits as Google Calendar: seamless integration, embedded workflow, higher adoption
- **Outcome 3:** Parity with Google Calendar feature; patient choice of calendar platform

---

## Scope

**In Scope:**
- OAuth 2.0 authorization flow via Microsoft identity platform
- Scope: Calendars.ReadWrite (create/update/delete events)
- Token storage: Securely store refresh token in patient account (encrypted)
- Consent UI: Clear explanation of permissions requested
- Revocation: Allow patient to disconnect Outlook calendar at any time
- Error handling: Clear messaging if authorization fails

**Out of Scope:**
- Automatic calendar selection (user selects during auth, defaults to primary if not specified)
- Multiple Outlook accounts per patient (one per patient account in Phase 1)
- Bi-directional sync (handled in US-009)
- Availability lookup from Outlook calendar

---

## Acceptance Criteria

1. **Authorization button:** Patient sees "Connect Outlook Calendar" button on booking confirmation and in settings/profile (parallel to Google Calendar button)
2. **OAuth flow initiated:** Clicking button redirects to Microsoft identity platform consent screen; PropellQ requests scope: `Calendars.ReadWrite`, `offline_access`
3. **Consent screen displays:** Standard Microsoft consent format: "PropellQ wants to: Manage your calendar events"
4. **Token handling:** Upon user consent, Microsoft returns authorization code; system exchanges code for access token + refresh token
5. **Secure token storage:** Refresh token encrypted with application key; stored in `PatientSessions.outlook_refresh_token` column
6. **Linked account confirmation:** After successful auth, system displays: "Outlook Calendar connected! Appointments will be added automatically." with link to "Manage Permissions"
7. **Disconnection:** Patient can click "Disconnect Outlook Calendar" in settings; system deletes refresh token and stops syncing
8. **Error handling:** If user denies scope or authorization fails, system displays: "Outlook Calendar authorization failed. Please try again or contact support."
9. **Both calendars supported:** Patient can have both Google AND Outlook calendars connected simultaneously; new appointments sync to both

## Technical Notes

**Requirement Mapping:** FR-004 (Calendar Integration - Google/Outlook)  
**Related Requirements:** NFR-005 (HIPAA - secure token storage), NFR-006 (secret management), TR-001, TR-003, TR-005  

**Technology Stack:**
- OAuth Library: Microsoft Authentication Library (MSAL) for .NET or Java
- Token Storage: Database encryption (AES-256), same as Google
- Redirect URI: `https://propellq.example.com/oauth/outlook/callback`
- Microsoft Graph API: `https://graph.microsoft.com/v1.0`

**Database Schema:**
```sql
ALTER TABLE PatientSessions ADD COLUMN (
  outlook_refresh_token VARCHAR (encrypted),
  outlook_access_token_expires_at DATETIME,
  outlook_calendar_id VARCHAR (nullable), -- e.g., "AQMkADAwATY..." or "primary"
  outlook_auth_status VARCHAR ('authorized', 'revoked', 'expired')
);
```

**OAuth Flow Pseudocode:**
```
GET /api/auth/outlook/authorize
  → Redirect to: https://login.microsoftonline.com/common/oauth2/v2.0/authorize
  ?client_id={APP_CLIENT_ID}
  &redirect_uri={CALLBACK_URL}
  &scope={SCOPES}
  &response_type=code

GET /api/auth/outlook/callback?code={AUTH_CODE}
  → Exchange code for tokens:
  POST https://login.microsoftonline.com/common/oauth2/v2.0/token
    client_id, client_secret, code, redirect_uri, scope
  ← Get: access_token, refresh_token, expires_in
  
  → Store tokens:
  UPDATE PatientSessions 
    SET outlook_refresh_token = ENCRYPT(refresh_token),
        outlook_auth_status = 'authorized'
  WHERE patient_id = {CURRENT_USER}
  
  → Redirect to: /profile/settings#calendar with success
```

**Authorization Scopes:**
- `Calendars.ReadWrite` - Create/modify/delete calendar events
- `offline_access` - Get refresh token for long-lived access

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Token stored unencrypted → security breach exposes patient's Outlook access | Critical | Use AES-256 encryption; store key in secure vault, not in code |
| Refresh token expires (rare but possible in long-term scenarios) → sync stops | Low | Monitor token status; proactive refresh before expiry; refresh on next API call |
| Patient has multiple Outlook accounts (work + personal) → confusion about which calendar | Low | During auth, display "Adding events to: [email@organization.com]"; allow user to select calendar from list |
| Microsoft Graph API returns 429 (rate-limited) → event creation fails | Medium | Implement exponential backoff; queue failed events for retry; monitor quota |
| OAuth consent screen times out → user redirected back with error | Low | Implement 15-minute timeout for OAuth session; display clear error message with retry link |
