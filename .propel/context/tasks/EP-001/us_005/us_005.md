# US-005: Implement Preferred Slot Swap Logic

**Epic:** EP-001 (Patient Appointment Booking)  
**Story Points:** 8  
**Priority:** CRITICAL  
**Status:** Not Started  
**Created:** 2026-06-17  

---

## User Story Statement

As a **patient**, I want to **automatically swap my current appointment to a preferred slot if that slot becomes available within 24 hours**, so that **I get my ideal appointment time without having to manually check the system or contact staff**.

---

## Business Value

- **Outcome 1:** Differentiator vs. competitors; "set-it-and-forget-it" UX increases customer satisfaction
- **Outcome 2:** Increases appointment flexibility; patients can book suboptimal slot knowing better option may appear
- **Outcome 3:** Reduces no-show rate; patients get preferred times → higher commitment

---

## Scope

**In Scope:**
- Background job: Monitor preferred slots hourly
- Swap logic: If preferred slot becomes available AND original slot still booked, automatically reschedule
- Notifications: Patient receives SMS + email within 5 minutes of swap with new appointment details
- Original slot release: Becomes available for re-booking immediately after swap
- Audit trail: Log all swap events with reason (preferred slot availability)
- Configurable timeout: Admin can set preferred slot monitoring window (24 hours default)

**Out of Scope:**
- Multiple preferred slot selections (only 1 preferred slot per booking in Phase 1)
- Automatic swap cancellation if patient books elsewhere during monitoring period
- Insurance re-validation after swap (assume insurance valid if original booking was approved)
- Cascading swaps (if swap destination has preferred slot) — out of scope

---

## Acceptance Criteria

1. **Preferred slot capture:** When patient books, system stores preferred_slot_id if provided during checkout (FR-002); database field `Appointments.preferred_slot_id` is nullable
2. **Background job frequency:** Swap monitoring job runs every 1 hour (configurable); checks all appointments with non-null preferred_slot_id and non-null swap_window_expires_at
3. **Swap eligibility check:** Job validates before swap:
   - Original appointment still exists and is not cancelled (status != "cancelled")
   - Preferred slot is now available (status = "available")
   - Preferred slot is still within 24-hour monitoring window (preferred_slot_id.datetime <= now + 24h)
   - No scheduling conflicts (preferred time doesn't overlap with other patient appointments)
4. **Automatic swap execution:** System updates original appointment: slot_id = preferred_slot_id, updated_at = now; marks original slot as available (status = "available"); sets preferred_slot_id = NULL, swap_window_expires_at = NULL
5. **Notification on swap:** Patient receives SMS + email within 5 minutes: "Great news! Your appointment has been moved to your preferred time: [date/time]. Original appointment details: [old_date/time]"
6. **Original slot availability:** Original slot returns to availability pool within 2 minutes of swap; system prevents re-booking during swap transaction (avoid race conditions)
7. **Audit logging:** Log entry created: appointment_id, event = "preferred_slot_swapped", original_slot_id, new_slot_id, triggered_by = "system", timestamp
8. **Edge case handling:** If preferred slot becomes unavailable before swap executes → job skips, appointment stays on original slot; no notifications sent

## Technical Notes

**Requirement Mapping:** FR-002 (Dynamic Preferred Slot Swap)  
**Related Requirements:** NFR-001 (99.9% uptime), NFR-005 (HIPAA), NFR-007 (logging), TR-006 (logging), TR-007 (tracing), TR-008 (resilience)  

**Technology Stack:**
- Background Job: .NET BackgroundService or Quartz scheduler
- Database: Transactional updates with BEGIN TRANSACTION / COMMIT to ensure atomicity
- Notification: Reuse email/SMS services from US-006
- Logging: Correlation ID tracking for swap events

**Database Schema:**
```sql
ALTER TABLE Appointments ADD COLUMN (
  preferred_slot_id INT FOREIGN KEY,
  swap_window_expires_at DATETIME,
  swap_completed_at DATETIME (nullable),
  swap_initiated_by VARCHAR (nullable) -- 'patient', 'system'
);

CREATE TABLE SlotSwaps (
  id INT PRIMARY KEY,
  appointment_id INT FOREIGN KEY,
  original_slot_id INT FOREIGN KEY,
  new_slot_id INT FOREIGN KEY,
  triggered_at DATETIME,
  completed_at DATETIME,
  status VARCHAR ('pending', 'completed', 'failed'),
  failure_reason VARCHAR (nullable)
);

CREATE INDEX idx_swap_monitoring ON Appointments (preferred_slot_id, swap_window_expires_at)
  WHERE status = 'booked' AND preferred_slot_id IS NOT NULL;
```

**Swap Job Pseudocode:**
```
FOREACH appointment WHERE preferred_slot_id IS NOT NULL 
  AND swap_window_expires_at > now:
  
  IF preferred_slot.status = 'available':
    BEGIN TRANSACTION
      UPDATE Appointments 
        SET slot_id = preferred_slot_id, 
            preferred_slot_id = NULL,
            swap_window_expires_at = NULL,
            swap_completed_at = now
      WHERE id = appointment.id
      
      UPDATE Slots 
        SET status = 'available' 
        WHERE id = appointment.slot_id (original)
      
      INSERT SlotSwaps (completed_at, status='completed')
      
      SEND NOTIFICATION (SMS + Email)
      INSERT AuditLog (event='preferred_slot_swapped')
    COMMIT
  ELSE
    INSERT AuditLog (event='swap_skipped_slot_unavailable')
```

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Swap job runs concurrently with user reschedule → data corruption | Critical | Use pessimistic locking; BEGIN TRANSACTION before checking/updating; test race conditions with concurrent reschedule attempt |
| Preferred slot monitoring window expires → patient doesn't get notification | Medium | Set swap_window_expires_at explicitly; if past expiry, skip swap but log for audit; send "swap window expired" email to patient |
| Swap job fails mid-transaction → slot locked but appointment not updated | High | Use ACID transactions; add circuit breaker; if job fails, alert admin; implement rollback retry logic |
| Patient receives duplicate swap notifications | Medium | Idempotent notification: check swap_completed_at before sending; include unique swap transaction ID in email |
| Swap happens 23:59:59 before window closes → patient tries to access original slot info | Low | Clear preferred_slot_id immediately after swap; redirect patient to new appointment in portal |
