# US-001: Search Appointments with Filters

**Epic:** EP-001 (Patient Appointment Booking)  
**Story Points:** 5  
**Priority:** CRITICAL  
**Status:** Not Started  
**Created:** 2026-06-17  

---

## User Story Statement

As a **patient**, I want to **search available appointment slots using multiple filters** (date, time, provider name, clinical specialty), so that **I can find an appointment that fits my schedule and clinical needs quickly**.

---

## Business Value

- **Outcome 1:** Reduces booking friction and enables self-service scheduling (removes 40% of front-desk calls)
- **Outcome 2:** Improves appointment discovery accuracy; patients find suitable providers on first attempt
- **Outcome 3:** Increases booking completion rate by making slot availability transparent and searchable

---

## Scope

**In Scope:**
- Search interface with filters: date range, time-of-day, provider name, clinical specialty
- Real-time slot availability display
- Provider profile preview (name, specialty, credentials, photo)
- Patient ability to save search filters (bookmark slot templates)
- Search results pagination and sorting

**Out of Scope:**
- Provider availability rules or scheduling logic (handled by admin)
- Insurance coverage validation (handled separately in EP-002)
- Telehealth vs. in-person filtering (out of Phase 1 scope)
- Advanced filtering (insurance, language, accessibility) - Phase 2

---

## Acceptance Criteria

1. **Search interface displays four filter categories:** date range picker, time range (morning/afternoon/evening), provider name autocomplete, clinical specialty dropdown
2. **Real-time availability:** System retrieves available slots from appointment database and displays them with <2 second latency on typical network
3. **Search results show:** Provider name, specialty, appointment date/time, location (clinic name), and "Book Now" button; all clickable to view provider details
4. **Filters are cumulative:** User can apply multiple filters simultaneously; results update in real-time without page reload
5. **No results message:** When no slots match criteria, system displays helpful message ("No slots available for selected filters; try extending date range or choosing different specialty") with suggestions
6. **Mobile-responsive layout:** Search interface is fully functional on 375px (mobile), 768px (tablet), and 1024px+ (desktop) breakpoints
7. **Accessibility compliance:** Filter labels linked to inputs; form is navigable via keyboard; screen reader compatible (WCAG 2.1 AA)

## Technical Notes

**Requirement Mapping:** FR-001 (Patient Self-Service Appointment Booking)  
**Related Requirements:** NFR-001 (99.9% uptime), NFR-002 (sub-500ms API latency p95), NFR-005 (HIPAA data handling)  

**Technology Stack:**
- Frontend: React (hook-based search component)
- API: .NET/Java REST endpoint: `GET /api/appointments/search?dateFrom=&dateTo=&specialty=&provider=`
- Database: Query optimized against `Appointments`, `Providers`, `Specialties` tables with indexing on status, date, specialty_id

**Data Retrieval:**
- Fetch only available slots (status = "available")
- Pagination: 10 results per page
- Sorting options: date (ascending), provider name (A-Z)

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| High volume of search queries during peak hours → API latency exceeds SLA | Medium | Implement Redis caching on search results; precompute common queries (top 5 specialties, next 30 days) during low-traffic windows |
| Provider/specialty data inconsistent between frontend filters and backend availability | Medium | Sync filter options to backend enum; validate filter values before query; add data validation test |
| Specialty filter shows deleted specialties | Low | Soft-delete specialties; exclude from search only when status = "deleted" |
