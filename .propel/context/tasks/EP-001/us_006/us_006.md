# US-006: Send Appointment Reminders (48h, 24h, 2h)

**Epic:** EP-001 (Patient Appointment Booking)  
**Story Points:** 5  
**Priority:** CRITICAL  
**Status:** Not Started  
**Created:** 2026-06-17  

---

## User Story Statement

As a **patient**, I want to **receive automated appointment reminders via SMS and email at 48 hours, 24 hours, and 2 hours before my appointment**, so that **I don't forget my appointment and can reschedule if needed before it's too late**.

---

## Business Value

- **Outcome 1:** Multi-channel reminders reduce no-show rate by 15%+ (primary lever for business goal)
- **Outcome 2:** SMS + email ensures high reach (95%+ SMS delivery, 99%+ email delivery)
- **Outcome 3:** Timezone-aware timing increases relevance; patient receives reminder at right time regardless of location

---

## Scope

**In Scope:**
- Scheduled reminder jobs: 48h, 24h, 2h before appointment
- SMS reminders: Concise message with appointment time, provider name, reschedule link
- Email reminders: Rich HTML template with appointment details, calendar attachment (ICS), reschedule/cancel links
- Patient preference customization: Allow patient to choose SMS only, email only, or both
- Retry logic: If SMS/email fails, retry up to 3 times with exponential backoff
- Delivery metrics: Track and report delivery success rates (SMS ≥95%, Email ≥99%)
- Timezone handling: Convert appointment time to patient's timezone

**Out of Scope:**
- Push notifications (mobile app not in Phase 1)
- Automated reschedule (just reminder + link to portal)
- WhatsApp/Messenger integration (future phase)
- Voice call reminders — out of scope

---

## Acceptance Criteria

1. **Reminder schedule:** System creates 3 reminder events at: 48 hours, 24 hours, 2 hours before appointment time (configurable intervals); job runs every 15 minutes to check for pending reminders
2. **SMS reminder content:** "Hi [FirstName], reminder: Your appointment with Dr. [Provider] is on [Date] at [Time]. [Reschedule Link]" (under 160 characters); sent via Twilio or similar SMS provider
3. **Email reminder content:** Rich HTML with appointment details, provider photo, location map, reschedule/cancel links, calendar attachment (ICS file); subject line: "Appointment Reminder: [Date] with Dr. [Provider]"
4. **Patient preferences stored:** `PatientPreferences.reminder_channels` = {'sms', 'email'} or {'sms'} or {'email'}; defaults to both if not set
5. **Timezone accuracy:** Reminder times calculated from patient's timezone (read from patient profile or browser); all times displayed in patient's local timezone
6. **Retry mechanism:** If initial send fails (SMS/email provider returns error), system retries up to 3 times: at +5 sec, +30 sec, +5 min; after 3 failures, flag for manual review
7. **Delivery tracking:** Log all reminder attempts in `ReminderLog` table: patient_id, appointment_id, reminder_type (48h/24h/2h), channel (sms/email), sent_at, delivery_status, retry_count
8. **Do-not-disturb handling:** If patient has opted out of all reminders (or cancelled appointment), system skips reminder creation

## Technical Notes

**Requirement Mapping:** FR-003 (Multi-Channel Appointment Reminders)  
**Related Requirements:** NFR-005 (HIPAA), NFR-007 (logging), TR-003 (API standards), TR-006 (logging), TR-007 (tracing)  

**Technology Stack:**
- Background Job: .NET BackgroundService or Quartz scheduler (runs every 15 minutes)
- SMS Service: Twilio (free tier: $0.0075/SMS)
- Email Service: SendGrid or Mailgun (reuse from US-004)
- Database: ReminderLog table to track delivery status
- Timezone Handling: Use `TimeZoneInfo` (.NET) or `moment-timezone` (Node.js)

**Database Schema:**
```sql
ALTER TABLE Appointments ADD COLUMN (
  reminder_sent_48h_at DATETIME (nullable),
  reminder_sent_24h_at DATETIME (nullable),
  reminder_sent_2h_at DATETIME (nullable)
);

CREATE TABLE ReminderLog (
  id INT PRIMARY KEY,
  appointment_id INT FOREIGN KEY,
  patient_id INT FOREIGN KEY,
  reminder_type VARCHAR ('48h', '24h', '2h'),
  channel VARCHAR ('sms', 'email'),
  sent_at DATETIME,
  delivery_status VARCHAR ('sent', 'failed', 'bounced'),
  retry_count INT,
  external_message_id VARCHAR (nullable), -- Twilio/SendGrid message ID
  failure_reason VARCHAR (nullable)
);

ALTER TABLE Patients ADD COLUMN (
  preferred_timezone VARCHAR DEFAULT 'America/New_York',
  reminder_channels JSON DEFAULT '{"sms": true, "email": true}'
);

CREATE INDEX idx_reminder_pending ON Appointments 
  WHERE status = 'booked' 
  AND DATEDIFF(minute, now(), appointment_time) 
  IN (48*60, 24*60, 2*60);
```

**Reminder Job Pseudocode:**
```
FOREACH appointment WHERE status='booked' AND appointment_time > now:
  
  -- Check 48-hour reminder
  IF (appointment_time - now <= 48 hours + 15 min buffer)
     AND reminder_sent_48h_at IS NULL:
    SCHEDULE_REMINDER(appointment, '48h')
    SET appointment.reminder_sent_48h_at = now
  
  -- Check 24-hour reminder
  IF (appointment_time - now <= 24 hours + 15 min buffer)
     AND reminder_sent_24h_at IS NULL:
    SCHEDULE_REMINDER(appointment, '24h')
    SET appointment.reminder_sent_24h_at = now
  
  -- Check 2-hour reminder
  IF (appointment_time - now <= 2 hours + 15 min buffer)
     AND reminder_sent_2h_at IS NULL:
    SCHEDULE_REMINDER(appointment, '2h')
    SET appointment.reminder_sent_2h_at = now

ASYNC FUNCTION SCHEDULE_REMINDER(appointment, type):
  IF patient.reminder_channels.includes('sms'):
    SEND_SMS(patient, appointment, type)
  IF patient.reminder_channels.includes('email'):
    SEND_EMAIL(patient, appointment, type)
```

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| SMS provider rate-limited or down → reminders delayed or fail | High | Use SMS provider with SLA guarantee (Twilio); set up alerts at 50% of quota; implement retry queue; fallback to email-only if SMS fails |
| Timezone calculation is wrong → patient receives reminder at wrong time | Medium | Test timezone conversion for UTC-8 to UTC+12; store timezone as enum; allow patient to override timezone in profile |
| Reminder sent to wrong phone/email → patient never receives it | Medium | Validate phone/email during intake (FR-005); send test SMS/email before appointment creation; provide "update contact info" link |
| Job sends duplicate reminders (job runs twice) → patient receives 2 SMS at same time | Low | Use appointment.reminder_sent_48h_at flag; job checks flag before sending; idempotent message ID tracking |
| Appointment cancelled but reminder still sent → patient confusion | Medium | Add appointment status check before reminder send; if cancelled, skip and clean up pending reminders |
