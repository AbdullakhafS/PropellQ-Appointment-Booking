# US-007: Authorize Google Calendar Integration

**Epic:** EP-001 (Patient Appointment Booking)  
**Story Points:** 5  
**Priority:** HIGH  
**Status:** Not Started  
**Created:** 2026-06-17  

---

## User Story Statement

As a **patient**, I want to **authorize the PropellQ application to access my Google Calendar via OAuth 2.0**, so that **my appointments are automatically added to my Google Calendar and I can manage them from my existing calendar app**.

---

## Business Value

- **Outcome 1:** Seamless integration with Google Calendar increases adoption; patient keeps appointments in one place
- **Outcome 2:** Reduces friction of switching between platforms; embedded workflow
- **Outcome 3:** Supports business goal of high booking completion and follow-through

---

## Scope

**In Scope:**
- OAuth 2.0 authorization flow (secure consent screen)
- Scope: calendar.events.create, calendar.events.update, calendar.events.delete (full appointment lifecycle)
- Token storage: Securely store refresh token in patient's account (encrypted in database)
- Consent UI: Clear explanation of permissions being requested
- Revocation: Allow patient to disconnect calendar at any time
- Error handling: Clear messaging if authorization fails (invalid credentials, scope denied, etc.)

**Out of Scope:**
- Automatic calendar selection (user selects target calendar during auth flow)
- Multiple calendar accounts (one Google calendar per patient account in Phase 1)
- Bi-directional sync implementation (handled in US-009)
- Calendar availability lookup (provider availability not exposed to Google Calendar)

---

## Acceptance Criteria

1. **Authorization button:** Patient sees "Connect Google Calendar" button on booking confirmation page and in settings/profile
2. **OAuth flow initiated:** Clicking button redirects to Google OAuth consent screen; PropellQ requests scopes: `calendar.events.create`, `calendar.events.update`, `calendar.events.delete`, `calendar.calendarList.read`
3. **Consent screen displays:** Clear messaging: "PropellQ wants to: Create and manage your calendar events" (Google-standard consent format)
4. **Token handling:** Upon user consent, Google returns authorization code; system exchanges code for access token + refresh token
5. **Secure token storage:** Refresh token encrypted with application key; stored in `PatientSessions.google_refresh_token` column (encrypted at-rest)
6. **Linked account confirmation:** After successful auth, system displays: "Google Calendar connected! Appointments will be added automatically." with option to "Manage Calendar Permissions"
7. **Disconnection:** Patient can click "Disconnect Google Calendar" in settings; system deletes `google_refresh_token` and stops syncing new events
8. **Error handling:** If user denies scope or authorization fails, system displays: "Google Calendar authorization failed. Please try again or contact support." with retry button

## Technical Notes

**Requirement Mapping:** FR-004 (Calendar Integration - Google/Outlook)  
**Related Requirements:** NFR-005 (HIPAA - secure token storage), NFR-006 (secret management), TR-001 (standards), TR-003 (API standards), TR-005 (secret handling)  

**Technology Stack:**
- OAuth Library: Google Auth Library for .NET or Java (e.g., `Google.Apis.Auth.OAuth2`)
- Token Storage: Database column with encryption (AES-256)
- Redirect URI: `https://propellq.example.com/oauth/google/callback`

**Database Schema:**
```sql
ALTER TABLE PatientSessions ADD COLUMN (
  google_refresh_token VARCHAR (encrypted),
  google_access_token_expires_at DATETIME,
  google_calendar_id VARCHAR (nullable), -- Primary calendar ID (e.g., "primary" or full email)
  google_auth_status VARCHAR ('authorized', 'revoked', 'expired')
);
```

**OAuth Flow Pseudocode:**
```
GET /api/auth/google/authorize
  → Redirect to: https://accounts.google.com/o/oauth2/v2/auth
  ?client_id={APP_CLIENT_ID}
  &redirect_uri={CALLBACK_URL}
  &scope={SCOPES}
  &response_type=code

GET /api/auth/google/callback?code={AUTH_CODE}
  → Exchange code for tokens:
  POST https://oauth2.googleapis.com/token
    client_id, client_secret, code, redirect_uri
  ← Get: access_token, refresh_token, expires_in
  
  → Store tokens:
  UPDATE PatientSessions 
    SET google_refresh_token = ENCRYPT(refresh_token),
        google_auth_status = 'authorized'
  WHERE patient_id = {CURRENT_USER}
  
  → Redirect to: /profile/settings#calendar with success message
```

**Authorization Scopes:**
- `https://www.googleapis.com/auth/calendar.events` - Create/modify/delete calendar events
- `https://www.googleapis.com/auth/calendar.calendarList.readonly` - Read list of calendars
- `https://www.googleapis.com/auth/calendar.settings.readonly` - Read calendar settings

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Refresh token stored unencrypted → token leaked in database breach | Critical | Use AES-256 encryption for all tokens at-rest; store encryption key in secure vault (not in code) |
| Token expires before user syncs appointment → calendar event not created | Medium | Implement token refresh logic before use; check expires_at and refresh if within 5 minutes of expiry |
| User changes password in Google → existing tokens invalidated | Low | Refresh token still valid; access token expires; implement refresh logic to handle expiry |
| User has multiple Google accounts → confusion about which calendar receives events | Low | During auth flow, display "These events will be added to: [selected calendar]"; allow user to select calendar from list |
| Google OAuth flow is interrupted (network error, user closes browser) → token not stored | Low | Session expires after 10 minutes; user must restart flow; graceful error handling |
