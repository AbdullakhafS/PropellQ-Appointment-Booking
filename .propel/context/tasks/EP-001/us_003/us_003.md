# US-003: Select Appointment & Lock Slot During Checkout

**Epic:** EP-001 (Patient Appointment Booking)  
**Story Points:** 8  
**Priority:** CRITICAL  
**Status:** Not Started  
**Created:** 2026-06-17  

---

## User Story Statement

As a **patient**, I want to **select an appointment slot and optionally choose a preferred but unavailable slot, then proceed through a secure checkout** (where my slot is locked from other patients), so that **my appointment is guaranteed and I can confidently proceed to payment or intake**.

---

## Business Value

- **Outcome 1:** Eliminates double-booking and race conditions where two patients book same slot
- **Outcome 2:** Enables "Preferred Slot Swap" feature (differentiator) by capturing both primary and preferred slots
- **Outcome 3:** Builds confidence in booking; patient knows their slot is locked and protected

---

## Scope

**In Scope:**
- Slot selection UI with visual feedback ("Selected", "Reserved")
- Optional "Preferred Slot Swap" section: let patient select an alternative preferred slot
- Checkout form: confirm appointment details, accept terms, submit booking
- Slot locking mechanism: mark slot as "reserved" during checkout (60-second timeout if user abandons)
- Clear error messaging if slot becomes unavailable during checkout
- Confirmation summary before final submission

**Out of Scope:**
- Insurance verification (handled in EP-002)
- Patient intake form (handled in EP-002)
- Payment processing (out of Phase 1 scope)
- Appointment modification/rescheduling (handled in EP-004)

---

## Acceptance Criteria

1. **Slot selection:** User clicks on calendar slot; slot is marked as "selected" (visual highlight); appointment details appear in right sidebar (date, time, provider, location)
2. **Preferred slot option:** Checkout form displays optional "Preferred Slot Swap" checkbox + dropdown to select alternative time; user can leave blank or choose preferred slot
3. **Slot locking mechanism:** Upon clicking "Confirm Booking", system transitions to checkout; selected slot is locked with status = "reserved" for 60 seconds (configurable); slot is visually removed from availability for other users
4. **Concurrent booking conflict:** If another user books the same slot before user confirms, system displays modal: "This slot was just booked. Please select another slot." and returns user to calendar (reservations cleared)
5. **Checkout form validates:** All required fields present (appointment date confirmed, provider confirmed, terms checkbox checked); error messages appear inline if fields incomplete
6. **Confirmation summary:** Before final submission, display: appointment date/time, provider name, location, estimated duration, and cost estimate (if applicable); user clicks "Confirm Booking" to finalize
7. **Mobile checkout UX:** Checkout form is single-column, touch-friendly on 375px width; all buttons are minimum 44px tall; form submits via POST to `/api/appointments/book`
8. **Accessibility:** Form labels linked to inputs; error messages announced to screen readers; keyboard navigation through all fields

## Technical Notes

**Requirement Mapping:** FR-001 (Patient Self-Service Appointment Booking), FR-002 (Dynamic Preferred Slot Swap)  
**Related Requirements:** NFR-005 (HIPAA), NFR-007 (logging), TR-003 (API standards)  

**Technology Stack:**
- Frontend: React form component with state management (Redux or Context)
- API: `POST /api/appointments/checkout` → creates reservation; `POST /api/appointments/book` → finalizes booking
- Reservation Table: Stores temporary locks with `slot_id`, `user_id`, `expires_at` (current_time + 60s)

**Database Changes:**
- Add `Reservations` table: slot_id (FK), user_id (FK), created_at, expires_at
- Add `preferred_slot_id` column to `Appointments` table (nullable, for future swap logic)
- Index: (slot_id, expires_at) for cleanup queries

**API Specification:**
```
POST /api/appointments/checkout
Request: { slotId, preferredSlotId (optional) }
Response: { reservationId, expiresAt, appointmentDetails }

POST /api/appointments/book
Request: { reservationId, preferredSlotId (optional) }
Response: { appointmentId, confirmationNumber, confirmationEmail }
Error Responses:
  - 409 Conflict: Slot unavailable (another user booked it)
  - 410 Gone: Reservation expired
```

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Race condition: Two users select same slot → both get through checkout | Critical | Implement database-level unique constraint on (slot_id, appointment_id); use pessimistic locking on slot during POST /book |
| User abandons checkout → reserved slot stays locked for 60s | Low | Automated cleanup job: delete Reservations where expires_at < now() every 30 seconds; send reminder email if user leaves checkout mid-process |
| Preferred slot swap selection not captured → future swap logic fails | Medium | Validate preferred_slot_id is non-null if preferred_slot_id column is non-nullable in schema; add unit test confirming swap data persists |
| Mobile form is hard to fill → user abandons before submission | Medium | Conduct UX testing with 5 patients on mobile; use browser autocomplete for patient name; reduce form fields to essentials in checkout |
