# US-004: Send Confirmation Email (PDF) Within 60 Seconds

**Epic:** EP-001 (Patient Appointment Booking)  
**Story Points:** 5  
**Priority:** CRITICAL  
**Status:** Not Started  
**Created:** 2026-06-17  

---

## User Story Statement

As a **patient**, I want to **receive an automated email confirmation with an attached PDF containing my full appointment details and instructions** within **60 seconds of booking**, so that **I have a record of my appointment and can share details with my family or take screenshots**.

---

## Business Value

- **Outcome 1:** Confirmations sent within SLA (60 seconds) increases patient confidence and reduces follow-up calls
- **Outcome 2:** PDF format is accessible; patient can download, forward, print without platform dependency
- **Outcome 3:** Email confirmation serves as contract; reduces disputes and supports compliance audit trail

---

## Scope

**In Scope:**
- Trigger confirmation email on successful booking (`POST /api/appointments/book` response)
- Generate professional PDF with: appointment date/time, provider name/specialty, location, confirmation number, cancellation/reschedule link, check-in instructions
- Email HTML template with: patient greeting, appointment summary, next steps (intake reminder if applicable), calendar attachment (ICS file), company contact info
- Verify email delivery success and retry failed sends (up to 3 retries)
- Store email metadata in audit log (sent_at, recipient, template_version)

**Out of Scope:**
- SMS confirmation (handled in US-006: Appointment Reminders)
- Patient's iCal/ICS auto-import (optional Phase 2)
- Encrypted email delivery (out of Phase 1)
- Multiple recipient addresses (family notifications) — Phase 2

---

## Acceptance Criteria

1. **Email triggered immediately:** Upon successful booking (HTTP 200 response from `POST /api/appointments/book`), background job enqueues confirmation email within 500ms; system responds to user before email is sent
2. **Email sent within 60 seconds:** Background job processes queue; email delivered to patient inbox within 60 seconds of booking (SLA: 95% of emails meet this target)
3. **PDF attachment contains:** Appointment date, time, provider name/specialty, location, confirmation number, duration, cost (if applicable), check-in instructions, cancellation/reschedule link with unique token
4. **Email template includes:** Professional greeting with patient name, appointment summary, next steps, "View in Portal" link, calendar attachment (ICS), company contact info, unsubscribe link
5. **Email subject is clear:** Subject line includes appointment date/provider name ("Your Appointment Confirmation - Dr. Smith on June 25, 2026")
6. **Delivery retry on failure:** If initial send fails, system retries up to 3 times over 10 minutes with exponential backoff (1s, 5s, 30s delays); after 3 failures, alert admin and flag in user's account for manual follow-up
7. **Audit logging:** All confirmation emails logged in `AuditLog` with: user_id, appointment_id, email_address, sent_at, delivery_status, template_version; searchable by user/appointment
8. **HIPAA compliance:** Email transmission encrypted (TLS); PDF doesn't contain patient SSN, full DOB; confirmation link is one-time-use token (expires after 30 days)

## Technical Notes

**Requirement Mapping:** FR-001 (Patient Self-Service Appointment Booking), FR-003 (Multi-Channel Reminders)  
**Related Requirements:** NFR-005 (HIPAA), NFR-007 (logging), TR-003 (API standards), TR-006 (logging standards)  

**Technology Stack:**
- Email Service: SendGrid, Mailgun, or free tier SMTP with retry logic
- PDF Generation: iText, pdfkit, or Chromium headless (headless-chrome)
- Background Job Queue: Redis + worker service (or .NET BackgroundService)
- Email Template: Handlebars or Liquid templating

**API/Database Changes:**
- Add `ConfirmationEmails` table: id, appointment_id, recipient_email, sent_at, delivery_status, template_version, retry_count
- Add `/api/appointments/{id}/confirmation/resend` endpoint for manual resend by user
- Audit log entry: appointment_id, event_type = "confirmation_email_sent", metadata = {delivery_status, template_version}

**Email Workflow:**
```
1. POST /api/appointments/book → 200 OK response
2. Enqueue job: { appointmentId, recipientEmail, templateVersion }
3. Worker polls queue; fetches appointment details from DB
4. Generate PDF using template + appointment data
5. Render HTML email template
6. Send via email service with attachment
7. Log result to AuditLog + ConfirmationEmails table
8. If failed: retry with exponential backoff
9. If all retries exhausted: alert admin (dashboard notification)
```

**PDF Template Fields:**
- Header: Company logo, "Appointment Confirmation"
- Patient Info: Name, DOB (last 4 digits only)
- Appointment Details: Date, time, provider, specialty, location, duration
- Provider Info: Photo, credentials, contact
- Next Steps: Intake reminder, check-in instructions, arrival time (15 min early)
- Footer: Confirmation number (scannable QR code), cancellation link, company contact

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Email service provider rate-limited → confirmations delayed beyond 60s SLA | High | Monitor SendGrid quota; set up alerts at 80% usage; implement backoff strategy; fallback to secondary email provider |
| PDF generation is slow → email delayed | Medium | Profile PDF generation; use cached templates; offload PDF generation to worker process (don't block HTTP response) |
| Patient email is invalid/typo → confirmation bounces | Medium | Validate email format during intake; send test email to address before booking; provide "resend confirmation" link in portal |
| PDF contains sensitive data (SSN, full DOB) → HIPAA violation | Critical | Review PDF template for PII; mask SSN; use last-4-digits for DOB; encrypt PDF if containing sensitive data |
| Email template not updated with new appointment types → patient confusion | Low | Store template_version in DB; test template rendering before deployment |
